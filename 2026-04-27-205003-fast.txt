  为什么以前不需要，现在需要？

  │   版本    │    NV12 数据来源    │   生存期管理    │    是否需要拷贝    │
  ├───────────┼─────────────────────┼─────────────────┼────────────────────┤
  │ USB       │ MppFrame（MPP       │ MPP             │ 不需要，MppFrame   │
  │ 摄像头 +  │ 内部维护独立帧池）  │ 自己管理，不随  │ 是独立生命期       │

  │ ISP       │ ISP 写入 V4L2       │ QBUF 后立即被   │ 必须拷贝，否则     │
  │ 直接输出  │ DMABUF 缓冲区       │ ISP 覆盖重写    │ Worker             │

  └───────────┴─────────────────────┴─────────────────┴────────────────────┘

  MppFrame 的数据在 MPP 内部有自己的内存池，不依赖 V4L2 缓冲区。而现在 ISP
  直接把数据写进 V4L2 的 DMABUF——它是个周转缓冲区，QBUF 还给驱动后下一秒就被 ISP
   新帧覆盖。所以必须在 DQBUF → 处理 → QBUF 这个窗口内把数据拷出来。

  那么"零拷贝"到底零在哪？

  这里有三条路径，只有一条免不了拷贝：

  ┌───────────┬───────────────────────────────────────┬─────────────────────┐
  │   路径    │                 方式                  │     是否零拷贝      │
  ├───────────┼───────────────────────────────────────┼─────────────────────┤
  │ ISP →     │ 硬件直接写入，CPU 不参与              │ 零拷贝              │
  │ DMABUF    │                                       │                     │
  ├───────────┼───────────────────────────────────────┼─────────────────────┤
  │ DMABUF →  │ RGA importbuffer_fd 直接从 DMABUF fd  │ 硬件加速，零 CPU    │
  │ BGR       │ 读取                                  │ 拷贝                │
  │ (画框)    │                                       │                     │
  ├───────────┼───────────────────────────────────────┼─────────────────────┤
  │           │ memcpy 从 DMABUF 映射区→堆内存，然后  │ 一次不可避免的 CPU  │
  │ NV12 →    │ Worker 中用 RGA                       │ 拷贝 + 硬件加速的   │
  │ NPU 输入  │ importbuffer_virtualaddr →            │ RGA 缩放            │
  │           │ importbuffer_fd 写入 NPU              │                     │
  ├───────────┼───────────────────────────────────────┼─────────────────────┤
  │ NPU 推理  │ rknn_create_mem 零拷贝绑定 + NV12     │                     │
  │ →         │ 直接送 MPP 编码                       │ 零拷贝              │
  │ 编码推流  │                                       │                     │
  └───────────┴───────────────────────────────────────┴─────────────────────┘

  有没有办法连这唯一一次 memcpy 也省掉？

  有——用 RGA 做 device-to-device 拷贝代替 CPU memcpy：

  DMABUF fd → RGA copy → 另一个"shadow" DMABUF buffer → Worker 直接
  importbuffer_fd

  这样 ISP 写完的 NV12 数据通过 RGA 硬件直接搬到一块独立的 DMABUF 里，完全绕过
  CPU。代价是多维护一套 shadow buffer 池和同步逻辑。对于现在的 1280x720 @
  30fps，memcpy 一次 NV12（~2.76 MB）的开销大约
  0.5-1ms，大概率不是瓶颈——所以你的做法是正确且务实的。

  ✶ Insight ─────────────────────────────────────
  理解了帧数据生存期管理这个核心约束：V4L2 DMABUF 是周转缓冲区（QBUF
  回收后立即被 ISP 覆盖），MppFrame
  是独立帧池。两种架构导致完全不同的数据传递策略。你的 malloc + memcpy
  不是设计妥协，而是对硬件 DMA 缓冲区生命周期的正确适配。
