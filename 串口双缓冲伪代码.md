## 一、核心数据结构

```
enum ObjClass { 塌方, 裂缝, 易燃易爆物, 高压线断裂 }

struct DetectionResult {
    ObjClass  label       // 障碍物类别
    float     x, y, z     // 三维坐标（相对于无人机，单位 m）
    float     confidence  // 置信度
    uint32_t  frame_id    // 帧序号（飞控用这个判断丢帧）
    uint64_t  timestamp   // 时间戳
}

struct SerialFrame {          // 定长，飞控好解析
    uint8_t   header = 0xAA  // 帧头标记
    uint8_t   frame_type     // 0=障碍物, 1=心跳/状态
    DetectionResult  det     // 检测结果
    uint8_t   checksum       // 异或校验
}
```

## 二、双缓冲（DoubleBuffer）— 保证 latest-value

```
class DoubleBuffer:
    buf : DetectionResult[2]  // 两片预分配 buffer
    front : atomic<int> = 0   // 读者当前可读的 buffer 索引

    // ===== NPU 线程调用（生产者） =====
    function write(det: DetectionResult):
        int next = 1 - front.load(acquire)  // 写备用 buffer
        buf[next] = det                     // 完整拷贝到备用区
        front.store(next, release)          // 原子翻转，公布新数据

    // ===== 串口发送线程调用（消费者） =====
    function read(out: DetectionResult) -> bool:
        int f = front.load(acquire)
        out = buf[f]         // 读当前公布的 buffer
        return true          // 永远有数据可读（读到最新的）
```

**关键保证**：写者和读者**永远不碰同一片 buf 索引**。`front` 翻转前，备用 buffer 已经完整写入。

## 三、小辅助队列（RingBuffer）— 串口抖动时缓存

```
template<T, N=4>            // N 很小，只缓存最近几帧
class RingBuffer:
    buf : T[N]
    w_idx : atomic<uint32_t> = 0
    r_idx : atomic<uint32_t> = 0

    // NPU 线程调用
    function try_push(det: T) -> bool:
        uint32_t w = w_idx.load(acquire)
        uint32_t r = r_idx.load(acquire)
        if ((w - r) >= N):         // 满了
            r_idx.store(r + 1, release)  // 主动丢弃最老的，腾位置
        buf[w % N] = det
        w_idx.store(w + 1, release)
        return true

    // 串口线程调用
    function try_pop(out: T) -> bool:
        uint32_t r = r_idx.load(acquire)
        uint32_t w = w_idx.load(acquire)
        if (r == w): return false   // 空
        out = buf[r % N]
        r_idx.store(r + 1, release)
        return true
```

**N=4 足够**：串口 5ms 轮询一次，NPU 20ms 产生一帧，一个周期内最多积压 1-2 帧。

## 四、串口发送线程 — 核心逻辑

```
function serial_send_loop():
    while running:
        det : DetectionResult

        // 策略：优先消费队列（缓冲帧），队列空了才读双缓冲
        if ring_buffer.try_pop(det):
            pass                  // 从队列取到历史缓存帧
        else:
            double_buffer.read(det)  // 队列空了，取最新帧

        // 组帧发送
        frame = SerialFrame {
            header: 0xAA,
            frame_type: 0,
            det: det,
            checksum: calc_xor(frame)
        }
        serial_write(frame)       // 非阻塞 write，失败下次重试
        sleep(5ms)                // 200Hz 发送轮询
```

**为什么队列优先于双缓冲？**

- 队列里的帧是有序的历史缓存，先发它们以免积压过期
- 队列空了说明发送端比产生端快，这时用双缓冲拿最新的，零等待
- 这就避免了 AI 代码中"队列永远不被消费"的问题

// STM32 串口中断接收
function UART_IRQHandler():
    byte = read_uart_byte()
    buffer.append(byte)

    // 帧同步：找 0xAA 帧头
    if buffer.size >= sizeof(SerialFrame):

            buffer.clear()

## 六、整体数据流

```
NPU 推理线程 (50fps, 绑 CPU4-7 大核)
    │
    ├──→ DoubleBuffer.write(det)      ← 覆盖写入，100ns 级
    └──→ RingBuffer.try_push(det)     ← 满则丢最老的，100ns 级
                                        （无锁，无等待）
    │
    ▼
串口发送线程 (200fps 轮询, 绑 CPU0-3 小核)
    │
    ├── RingBuffer.try_pop()?  ← 优先消费队列
    └── DoubleBuffer.read()    ← 队列空则读最新
    │
    ▼
serial_write(SerialFrame)  →  UART TX  →  飞控
```

## 七、和 AI 代码的差异总结

| 对比项 | AI 代码 | 本设计方案 |
|--------|---------|-----------|
| 双缓冲 | 假的单 buffer + flag，有 data race | 真双 buffer + atomic index 翻转 |
| 队列优先级 | latest 优先，队列永远不被消费 | 队列优先消费，latest 兜底 |
| 队列满策略 | 静默丢新数据 | 主动丢最老的腾位置 |
| 串口发送 | 绑推理线程同一核 | 独立绑小核，不干扰推理 |
| 帧同步 | 无 | 0xAA 帧头 + checksum |
