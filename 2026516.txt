第2题:在无人机项目中，你提到了基于V4L2+DMABUF的零拷贝方案。请详细说明从IMX415采集 RAW 数据到NPU推理，数据是如何在内存中流转的?为什么要强调“零拷贝”,它解决了什么核心痛点?
这个零拷贝方案的核心其实就是利用了 Linux 的 DMABUF 机制，让数据在各个硬件单元之间“只传指针不传数据”。

具体的数据流转是这样的：首先，我通过 V4L2 缓冲区类型选为dmabuf格式，在内核态预分配了一块 CMA 连续物理内存。当 IMX415 采集到 RAW 数据后，由 ISP 硬件直接将处理好的 YUV 数据写入这块内存。关键点在于，ISP输出所在的物理内存的文件描述符，我们在分配物理内存时就拿到了。
接下来，我把这个 FD 直接传递给 RGA 硬件加速模块，让它完成 YUV 到 RGB 的转换和缩放。RGA 处理完后，输出为NPU输入的物理内存，并且我们提前为NPU的输出侧也分配了物理内存。整个链路中，CPU 仅仅是负责在用户态传递 FD 以及下发硬件指令，完全没有参与像素级的数据搬运。
为什么要费劲做这个？核心痛点就是CPU 负载。
不使用dambuf直接传递物理内存的话，每次推理前需要memcpy输入数据到 NPU 内存，推理后再 memcpy 结果回 CPU 内存。两张额外的内存拷贝在 30FPS 下就是每秒 60 次拷贝：
输入：1280x720 NV12→640x640 RGB 已由 RGA 完成（不在比较范围内）
但输入 memcpy 是把 RGA 结果从 CPU 内存拷到 NPU 内存：640x640x3 ≈ 1.2MB，每秒 30 次 ≈ 36MB/s
输出 memcpy 是把三个输出分支拷回 CPU 内存，数据量小得多但也是额外开销

第6题:JD中提到负责系统监控功能。如果一个C++编写的后台服务突然CPU占用率达到100%，你会使用哪些Linux工具进行排查?请给出从定位线程到定位具体代码行的完整思路。
在不修改代码的前提下，我会直接使用 `top -H -p [PID]` 命令，这个 `-H` 参数能让我看到该进程下所有线程的实时资源消耗，我会迅速锁定那个 CPU 占用接近 100% 的线程 ID（TID）。 拿到 TID 后，我会使用 `gdb attach [PID]` 挂载到正在运行的服务上。进入 GDB 环境后，先执行 `info threads` 列出所有线程，找到刚才记录的 TID 对应的 GDB 线程编号。接着通过 `thread [编号]` 切换到目标线程，直接输入 `bt`（backtrace）打印当前的函数调用栈。因为 CPU 满载通常意味着程序正卡在某个死循环或极高频的计算逻辑中，通过 `bt` 看到的栈顶函数和对应的源码行号，基本就能定位到具体是哪一行代码在‘空转’。 如果调用栈看起来很正常，可能是某些逻辑触发了频繁的系统调用，我会再配合 `perf top -p [PID]` 观察一下，看哪些函数占用了最多的 CPU 周期，这样能更直观地发现热点函数，而不仅仅是停留在某一个瞬间的堆栈快照上。
