步骤A：批量提交任务到线程池（控制并发度）
从「读队列（g_readQueue）」中获取未处理的帧，提交到 npu_pool（线程池）并行推理，同时通过 MAX_CONCURRENT_FRAMES 控制并发任务数，避免内存溢出（图像数据占用内存较大，并发过多会导致OOM）：

// 1. 定义存储从读队列取出的帧数据
FrameData inputFD;
// 2. 条件判断：读队列非空 + 当前并发任务数 < 最大并发数（避免内存溢出）
if(!g_readQueue.empty() && tasks_inflight.size() < MAX_CONCURRENT_FRAMES)
{
    // 3. 从读队列取出一帧（线程安全，由 SafeQueue 内部锁保证）
    if(g_readQueue.dequeue(inputFD))
    {   
        // 4. 提交任务到线程池：获取任务结果的 future（“结果凭证”）
        auto future = npu_pool.submit_task_async(inputFD.index, inputFD.frame);
        // 5. 将“帧索引→future”存入 tasks_inflight，管理并行任务
        tasks_inflight[inputFD.index] = std::move(future); // 注意：future不可拷贝，必须用std::move
        // 6. 日志输出：打印当前并发任务数和处理的帧索引（调试用）
        std::cout << "当前并发任务数: " << tasks_inflight.size() 
                  << "，处理到第 " << inputFD.index << " 帧" << std::endl;
    }
}
关键逻辑：MAX_CONCURRENT_FRAMES（如代码中定义为10）是并发控制阈值——当 tasks_inflight.size() 达到10时，不再从读队列取帧，避免线程池任务过多导致内存占用激增（每帧图像按1920×1080×3计算，单帧约6MB，10帧即60MB，符合内存安全范围）；
这是我们之前旧架构对MAX_CONCURRENT_FRAMES的解释，但我现在有疑问，即使控制了向线程池交付图像，但对应的图像资源不还是堆积在读线程的g_SafeQueueRead中吗
这是一个极其刁钻且极具深度的架构拷问！你非常敏锐地察觉到了流水线并发设计中经典的**“挤水球效应（Water Balloon Effect）”**——在这里捏紧了，压力只会转移到另一个地方鼓起来。

你的质疑**完全正确**！如果仅仅依靠 `MAX_CONCURRENT_FRAMES` 控制提交给线程池的任务，那些来不及处理的图像确实会全部堆积在读线程的 `g_SafeQueueRead` 中。

为了让你在面试中能把这个底层逻辑讲得更透彻，我们来拆解一下旧架构的真实运转情况，以及它为什么会引发“灾难”，最后看看新架构是如何彻底解决这个痛点的。

### 一、 旧架构的“两道防线”与隐藏危机

在旧架构中，为了防止 OOM（内存溢出），其实存在**两道防线**（尽管第二道防线治标不治本）：

1. **第一道防线（你贴出的代码）：** `MAX_CONCURRENT_FRAMES`。它保护的是**线程池和 NPU 的运算内存**，确保同时只有 10 帧图像被 `future` 引用并处于活跃处理状态。
2. **第二道防线（隐藏的队列上限）：** `g_SafeQueueRead` 作为生产者-消费者队列，通常在初始化时会设置一个最大容量（例如我们在之前文档中看到的 `maxSize = 100`）。
   * 当 `ReadVideo` 疯狂采集，导致队列里塞满 100 帧时，队列的内部锁（`std::condition_variable`）会**强行阻塞（Block）** `ReadVideo` 线程。
   * 也就是说，`ReadVideo` 会卡在 `enqueue` 这一步，无法再调用 `malloc` 申请新内存，从而勉强保住了系统没有发生 OOM。

#### 💣 旧架构的致命灾难：Bufferbloat（缓存膨胀）
虽然依靠队列上限没有 OOM，但这带来了一个音视频领域最致命的问题：**极其恐怖的延迟（Latency）！**
试想一下，如果队列里堆积了 100 帧 1080P/720P 的图像：
* **内存占用：** 100 帧 × 1.38MB ≈ 138MB 的纯堆内存浪费。
* **延迟爆炸：** 摄像头每秒采集 30 帧。当第 100 帧进入队列排队，等到它终于从队列里出来，被 Worker 拿到并推流到屏幕上时，**时间已经过去了整整 3.3 秒！**（100 / 30fps）。
对于一个实时视频流系统来说，3.3 秒的画面延迟是绝对无法接受的工程事故。

### 二、 新架构是如何从根源斩断“堆积”的？

在新架构中，我们彻底抛弃了这种“靠长队列硬扛”的笨重思路，引入了高级流控机制：**背压（Backpressure）与物理源头限流**。

1. **从“虚拟堆积”变为“物理受限”：**
   我们在底层预分配了 CMA 连续物理内存池（例如只有 4 块 `dmabuf_fd`）。这就意味着，全系统最多只能同时存在 4 帧图像。**连堆积的物理空间都不存在，自然就不可能 OOM。**
2. **主动丢帧（Drop Frame）取代死锁阻塞：**
   当 `ReadVideo` 拿到一帧新画面，准备顺着“静态路由”分配给 0 号 Worker 时，它会去检查 `worker_ctx[0].is_busy`。
   * 如果为 `true`（说明 NPU 还没处理完上一帧），`ReadVideo` **绝不会**像旧架构那样把一兆多的像素塞进队列死等。
   * 此时 `ReadVideo` 会直接启动**主动丢帧机制**，只向队列里塞入一个仅有几个字节的空壳信号（`skip_inference = true`），而这帧画面的物理内存则直接被摄像头下一帧的新数据无情覆盖。

### 💡 面试升华话术（降维打击）

如果面试官问：“你旧架构里做了并发控制，但队列不会堆积内存吗？”

你可以这样回答：
> “您说得非常对，这也正是我重构架构的核心驱动力之一。
>
> 在旧架构中，即便限制了 inflight 数量，由于采集端（生产者）速度大于推理端（消费者），数据不可避免地会堆积在 `SafeQueue` 中。哪怕限制了 Queue 的深度（比如 100 帧）防止了 OOM，但这会引发严重的 **Bufferbloat（缓存膨胀）**，导致视频推流产生几秒钟的物理延迟。
> 
> **为了根治这个问题，我在新架构中引入了源头背压（Backpressure）机制。** 我利用底层的 CMA 物理内存池和 Worker 私有的 `is_busy` 原子状态锁，将原本在软件队列中的‘被动阻塞等待’，优化成了硬件物理层面的‘主动丢帧策略’。
> 如果下游 NPU 遇到算力瓶颈，采集端会立刻丢弃帧的像素数据，仅向汇聚线程传递轻量级的保序占位符。这不仅彻底消除了队列堆积带来的 OOM 风险，更把系统的端到端延迟死死压制在了百毫秒级以内，实现了真正的实时音视频流转。”
