无人机灾区环境风险评估 使用开发板：radxa5c（3588S2）
项目简介： 本项目为无人机自主避障系统提供视觉感知基础，完成目标检测模块部署。通过部署yolo 算法，在 RK3588
开发板实现实时障碍物检测（塌方、裂缝、易燃易爆物、高压线断裂）等危险区域。检测结果通过标准化接口传输至飞
控，实现避障决策。
主要工作：
IMX415-MIPI 驱动移植、零拷贝方案：完成 IMX415 RAW10 数据采集到 YOLOv5 RGB 输入的全链路打通：配置
MIPI CSI-2 四通道传输，通过 ISP 实现 RAW 转 YUV420，利用 RGA 硬件加速 YUV 转 RGB（6ms / 帧），建立 sensor
到 NPU 的零拷贝数据通路
低延迟视频流采集：基于 V4L2 + DMABUF 框架，预分配 CMA 物理连续内存池（4×1.38MB 循环缓冲），poll +
DQBUF事件驱动采集，单帧采集延迟 ≤17ms。
硬件加速处理流水线：RGA 预处理：调用 RK3588 RGA 模块执行 YUV420→RGB565 转换（6ms / 帧，软件实现需
30ms）；NPU 推理：调用 3 核 NPU 加速 YOLOv5s 推理，吞吐量 40FPS（单核仅 12FPS）；VPU 硬件编码：采用
MPP 硬编码 H.264（1080P@30FPS，CPU 占用率 5%），FFmpeg 推流端延迟小于 200ms。
高并发软件架构：基于 C++11 实现三阶段高并发流水线（采集 → 调度推理 → 编码推流）。采集线程通过 Atomic 标志检测下游负载，忙时主动跳过帧写入实现背压控制；调度线程(Aggregator)采用 Per-Worker 独立队列 + 轮询分发代替全局任务队列，消除锁竞争，并通过非阻塞结果收集 + map 按序重排保证帧序；推理线程池(4个Worker)各自持有独立 NPU 实例，专司推理与绘制，阶段间用双条件变量有界队列隔离。

**版本三：侧重技术深度的扩展版（~3 分钟，适合技术面）**

> 面试官你好，我叫 [姓名]，应聘嵌入式软件开发岗。
>
> 最近一年我主要在做 Rockchip RK3588 平台上的嵌入式实时视频分析，项目是为无人机自主避障系统提供视觉感知能力，在 Radxa 5C 开发板上实现从 IMX415 摄像头采集到 YOLOv5s 推理再到 RTMP 推流端到端流水线。
>
> 具体工作分三块。第一块是视频采集链路，IMX415 输出 RAW10 通过 MIPI CSI-2 四通道进入 ISP，由 RKAIQ 3A 服务完成自动曝光和白平衡，ISP 输出 NV12 格式。我基于 V4L2 MPLANE 接口和 DMABUF 内存模式，从 CMA 分配器预分配了 4 块 1.38MB 物理连续内存作为循环缓冲池，用 poll 加 DQBUF 事件驱动采集，保证单帧采集延迟在 17ms 以内。
>
> 第二块是推理加速。我部署了 YOLOv5s 模型，利用 RGA 硬件做图像预处理，包括 NV12 到 RGB 的格式转换和缩放到 640×640，这些通过 RGA 硬件只需要 6ms，软件实现要 30ms。最关键的是我打通了从 sensor 到 NPU 的零拷贝通路：RGA 直接从 ISP 的 DMABUF 读取数据，缩放转换后直写 NPU 的物理内存，全程不走 CPU 拷贝。推理时三核 NPU 并行，吞吐量达到 40FPS，单核只有 12FPS。推理结果通过 MPP 硬编码 H.264，CPU 占用率只有 5%。
>
> 第三块是软件架构。我基于 C++11 实现了一个三阶段高并发流水线（采集 → 调度推理 → 编码推流），有三个设计要点。**一是采集端的背压控制**：ReadVideo 线程通过 Atomic 标志检测下游 Worker 是否繁忙，忙时跳过该帧的 NPU 写入直接释放，防止流水线阻塞。**二是调度层的无锁分发**：Aggregator 线程采用 Per-Worker 独立队列加轮询调度代替传统的全局任务队列，每个 Worker 持有自己的队列和互斥锁，提交任务时只锁目标 Worker，消除了所有线程争一把全局锁的热点。**三是调度层的非阻塞收拢**：Worker 完成后结果入公共队列，Aggregator 非阻塞轮询批量收拢，乱序到达的结果按帧索引入 map 缓存后顺序写出，避免 head-of-line blocking。
>
> 以上就是我项目的主要工作，谢谢。
