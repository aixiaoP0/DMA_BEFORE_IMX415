## 推荐的结合方式

不要把 MPP 换成 x264。RK3588 上 MPP 硬编码更符合嵌入式实时系统定位，x264可以作为 PC/虚拟机的兼容编码器或模拟源。

在 MPP 输出之后增加一个“编码帧分发层”：

```text
旧架构原有管线
V4L2 → 推理线程池 → 按序输出 → MPP编码
                              │
                              ▼
                      H264 EncodedFrame
                              │
                 ┌────────────┼────────────┐
                 ▼            ▼            ▼
              RTMP Sink     TCP Sink     RTP/UDP Sink
                 │            │            │
                 └────────────┴────────────┘
                              ▼
                    虚拟机 C++ Client
                              │
       接收 → Jitter Buffer → FFmpeg解码 → OpenGL/ImGui
```

关键原则是：一帧只进行一次 MPP 编码，多个协议共享编码后的 H.264 Access Unit，不能为每种协议重复编码。

## 需要怎样改现有代码

当前主要耦合点是：

```cpp
MPP编码完成 → write_frame() → 直接RTMP发送
```

建议拆成：

```cpp
MPP编码完成
    → on_encoded_frame(EncodedFrame)
    → EncodedFrameDispatcher
        → RtmpSender
        → TcpSender
        → RtpSender
```

`EncodedFrame` 至少应包含：

```cpp
struct EncodedFrame {
    std::shared_ptr<std::vector<uint8_t>> data;
    uint64_t frame_id;
    int64_t capture_timestamp_us;
    int64_t encode_done_timestamp_us;
    bool key_frame;
    uint32_t pts;
};
```

每个发送端使用独立的有界线程安全队列。否则 TCP 慢客户端或网络阻塞会卡住 MPP 编码回调，最终拖慢整条推理管线。

## 三种协议如何实现

- RTMP：保留现有实现，作为兼容性和直播分发链路。
- TCP：增加消息头，如 `长度 + 帧号 + 时间戳 + flags + H.264数据`；处理粘包、拆包和慢客户端背压。
- UDP：如果只是自己设计分片协议，适合学习，但工程上与 RTP 功能重复。
- RTP：解析 Annex-B、按 NALU 发送；大 NALU 使用 FU-A 分片，维护 sequence、90 kHz timestamp、marker bit，并通过 SDP传递编码参数。

TCP 发送队列满时不能无限堆积。实时视频通常丢弃旧 P 帧，等待下一个 IDR 重新同步，而不是积累数秒延迟。

## 客户端可以补齐的能力

虚拟机中不再仅使用 `ffplay` 或 FFmpeg 命令，而是实现：

```text
NetworkReceiver
    → 重组/Jitter Buffer
    → FFmpeg avcodec_send_packet/receive_frame
    → YUV上传纹理
    → OpenGL shader转换/渲染
    → ImGui展示指标
```

可以展示：

- 接收、解码、渲染 FPS
- 接收码率
- RTP 丢包率、乱序率
- Jitter Buffer 深度
- NACK 请求和恢复数量
- FEC 恢复率
- 端到端延迟 p50/p95/p99

跨设备统计单向端到端延迟时，板端和虚拟机必须进行 NTP/PTP 时钟同步；否则只能准确统计 RTT，不能直接用两台机器的本地时间相减。

## FEC/NACK建议顺序

不要一开始同时实现所有弱网功能，合理顺序是：

1. MPP输出帧抽象和分发层
2. TCP客户端
3. RTP FU-A分片、重组和SDL/SDP参数
4. FFmpeg解码
5. OpenGL/ImGui显示
6. Jitter Buffer
7. 丢包模拟与延迟统计
8. NACK和发送端重传缓存
9. 最后增加FEC

FEC和NACK解决的是不同问题：NACK消耗至少一次往返时间，适合低RTT；FEC无需等待重传，但会增加带宽。
