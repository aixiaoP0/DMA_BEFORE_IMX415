## 5.1 Aggregator 要解决什么问题

线程池中有 3 个 Worker 并行推理，每个 Worker 处理速度不同：

```
ReadVideo 采集顺序:  frame0  frame1  frame2  frame3  frame4  frame5 ...
                        │       │       │       │       │       │
                        ▼       ▼       ▼       ▼       ▼       ▼
                    Worker2  Worker0  Worker1  Worker3  Worker0  Worker2  ...
                    完成快    完成慢    完成快    完成中    完成快      ...
                        │       │       │       │       │       │
                        ▼       ▼       ▼       ▼       ▼       ▼
                    结果顺序:  frame0  frame2  frame3  frame1  frame4  ...
```

如果没有保序机制，画面会出现**帧序跳跃**——先完成的帧反而先被编码推流，视频画面前后颠倒。

> **关键理解：** Aggregator 本质上是一个**乱序执行 + 顺序提交引擎**。它允许 Worker 乱序完成推理（充分利用并行），但只按原始帧序写入输出队列。这个模式在数据库领域叫"out-of-order execution with in-order commit"——与 CPU 的 Tomasulo 算法有相同的设计哲学。

---

## 5.2 数据结构：`map<int, ProcessResult> result_buffer`

```cpp
map<int, ProcessResult> result_buffer;
int nextWriteIndex = 0;
```

这是整个保序机制的核心：

| 变量 | 类型 | 作用 |
|------|------|------|
| `result_buffer` | `map<int, ProcessResult>` | key=帧序号, value=已完成的结果缓存（map 有序） |
| `nextWriteIndex` | `int` | 期望下一个写入输出队列的帧序号，单调递增 |

**为什么用 `map` 而不是 `unordered_map`？**

- map 是有序容器，`find(nextWriteIndex)` 能找到当前应写入的帧
- 如果某个帧被跳过（`skip_inference`），map 的迭代器遍历有序，便于调试
- `O(log n)` 对于最多 20 个元素完全够用

```
result_buffer = {
    0: ProcessResult,    // 已完成，等待写入（nextWriteIndex=0 时立即处理）
    1: ProcessResult,    // 已完成
    2: ...               // 推理中，尚未到达
    3: ...               // 推理中
}

nextWriteIndex = 0  // 只检查 key == 0 的结果
```

> **与旧版 `future` 方案对比：** 旧版 Aggregator 通过 `submit_task_async()` 返回 `future<ProcessResult>` 存入 `map<int, future>`，通过 `wait_for(0)` 轮询完成状态。新版使用 `submit_to_worker()`（无返回值）+ 独立 `result_queue` + `map<int, ProcessResult> result_buffer` 重排，消除了每次提交的 promise 堆分配和 future 原子同步开销。

## 5.3 主循环：三个阶段

### 步骤 A：提交循环

```cpp
while (!g_SafeQueueRead.empty() && tasks_inflight.size() < 20) {
    if (g_SafeQueueRead.dequeue(inputFD)) {
        int target_worker = inputFD.index % g_num_workers;

        if (inputFD.skip_inference) {
            // 丢帧标记：创建空白结果维持保序
            ProcessResult blank;
            blank.success = false;
            tasks_inflight[inputFD.index] = std::move(blank);
        } else {
            gthreadpool.submit_to_worker(
                target_worker,
                inputFD.index, inputFD.frame,
                inputFD.width, inputFD.height);
            tasks_inflight[inputFD.index] = ProcessResult(); // 占位
        }
    }
    is_idle = false;
}
```

关键行为：

- **Round-Robin 分发**：`index % g_num_workers` 确定目标 Worker
- **`is_busy` 已在 ReadVideo 中处理**：如果 Worker 的 NPU 缓冲区被占用（仍在推理上一帧），ReadVideo 已标记 `skip_inference=true`，此处直接创建空白结果
- **限流**：`tasks_inflight.size() < 20` — 最多 20 帧在途，~~防止内存溢出~~
- 限制 20 实际上是保护 g_SafeQueueWrite 不被一次冲爆的工具，而不是保护内存的工具。 
- **不等待**：`dequeue` 是非阻塞的（SafeQueue 队列为空返回 false）

### 步骤 B：收集循环

```cpp
// 从结果队列批量取出已完成的结果
WorkerResult wr;
while (gthreadpool.try_get_result(wr)) {
    result_buffer[wr.index] = std::move(wr.result);
}

// 按序写出
auto it = result_buffer.find(nextWriteIndex);
while (it != result_buffer.end()) {
    outputFD.index = nextWriteIndex;
    outputFD.frame = it->second.processed_img;       // 引用计数浅拷贝
    outputFD.nv12_data = it->second.nv12_data;

    g_SafeQueueWrite.enqueue(outputFD);

    result_buffer.erase(it);
    nextWriteIndex++;
    it = result_buffer.find(nextWriteIndex);
}
```

关键行为：

- **批量收集**：`try_get_result` 非阻塞地从结果队列 drain 所有已完成结果
- **`skip_inference` 帧自动衔接**：空白 `ProcessResult` 也会出现在 `result_buffer` 中，`nextWriteIndex` 正常推进，不会卡死
- **连续收集**：用 `while` 而不是 `if` — 如果 frame 5 完成后恰好 frame 6 也完成了，一口气收两个
- **结果缓冲区**：`result_buffer` 是一个临时重排区，最多缓存 20 个已完成的结果

> **对比旧版：** 旧版使用 `future.wait_for(0)` 逐个检查 nextWriteIndex 的完成状态，新版通过 `result_queue + try_get_result` 批量 drain，然后用 `map` 按序重排。新模式消除了每次 submit 一次 promise 堆分配和 wait_for 轮询开销。

### 步骤 C：退出检测（第 367-371 行）

```cpp
if (g_readFinish && g_SafeQueueRead.empty() && tasks_inflight.empty()) {
    break;
}
```

退出条件三要素**缺一不可**：

| 条件 | 说明 |
|------|------|
| `g_readFinish == true` | ReadVideo 已停止采集 |
| `g_SafeQueueRead.empty()` | 读队列中的残留帧已全部取出 |
| `tasks_inflight.empty()` | 所有已提交的推理任务都已完成并收集 |

**退出后：**

```cpp
g_processFinish = true;  // 通知 WriteVideo 线程可以退出了
```

## 5.4 空闲检测与 CPU 节流

```cpp
if (is_idle) {
    std::this_thread::sleep_for(std::chrono::milliseconds(1));
}
```

- `is_idle` 初始化为 `true`，只有步骤 A 或步骤 B 有实际工作时才设为 `false`
- 既没有新帧可提交、也没有已完成帧可收集时，线程休眠 1ms 避免空转

## 5.5 skip_inference 丢帧处理

当 ReadVideo 检测到目标 Worker 的 `is_busy=true`（NPU 缓冲区正在被该 Worker 使用），会设置 `skip_inference=true`：

```cpp
// ReadVideo 中
if (gthreadpool.is_worker_busy(target_worker)) {
    frame_temp.skip_inference = true;  // 标记丢帧
} else {
    gthreadpool.set_worker_busy(target_worker, true);
    // Path C: RGA 直写 NPU 物理内存
    // ...
}
```

Aggregator 中需要保证丢帧不影响保序：

```cpp
if (inputFD.skip_inference) {
    ProcessResult blank;
    blank.success = false;
    result_buffer[inputFD.index] = std::move(blank);
    // 空白结果占位，nextWriteIndex 正常推进
}
```

WriteVideo 中 `nv12_data == nullptr` 的帧被自然跳过，不影响 FPS 统计。

> **追问：** `skip_inference` 帧是否真的有必要入队？
> → 必要。如果直接丢弃，`nextWriteIndex` 会永远等不到缺失的帧序，整个输出管线卡死。`skip_inference=true` + 空白 ProcessResult 是维持保序连续性的关键。

## 5.6 保序执行时序演示

假设 Worker 处理帧的速度不同：

```
时间 →

提交:   0  1  2  3  4  5          (Aggregator 从读队列取帧，round-robin 分发)
        │  │  │  │  │  │
        ▼  ▼  ▼  ▼  ▼  ▼
        W0 W1 W2 W3 W0 W1        (round-robin: index % 4)

完成:    2  0  1  5  3  4         (实际完成顺序不固定，通过 result_queue 回传)
         │  │  │  │  │  │
         ▼  ▼  ▼  ▼  ▼  ▼

result_buffer（收集循环 drain 全部已完成结果）:
  结果 0 → 等待 nextWriteIndex
  结果 1 → 等待 nextWriteIndex
  结果 2 → 等待 nextWriteIndex
  ...

nextWriteIndex=0:  result_buffer 中有帧 0 → 写入，nextWriteIndex=1
nextWriteIndex=1:  result_buffer 中有帧 1 → 写入，nextWriteIndex=2
nextWriteIndex=2:  result_buffer 中有帧 2 → 写入，nextWriteIndex=3
nextWriteIndex=3:  result_buffer 中无帧 3 → 等待下一轮收集
```

> **追问：** 如果某一帧推理特别慢（比如 frame 5），会怎样？
> → 后面的帧即使先完成也必须等待 frame 5，造成 **Head-of-Line blocking**（队头阻塞）。这是保序的固有代价。可通过 `skip_inference` 主动丢帧缓解。

## 5.8 result_queue 配合机制

Aggregator 不直接与 Worker 通信，它通过 `result_queue（结果收集队列）` 批量获取结果：

```
Aggregator:                         Worker[i]:
   submit_to_worker(index, ...)        ProcessResult res
       │                               result_queue.push({index, res})
       │                               worker_ctx[id].is_busy = false
       │   
   try_get_result() — 非阻塞批量 drain
       │   
   result_buffer[index] = std::move(result)
       │   
   find(nextWriteIndex) → 按序写出
```

| 操作 | 位置 | 作用 |
|------|------|------|
| `submit_to_worker(target, index, img, w, h)` | `thread_pool.cpp:82-99` | 直接入队目标 Worker 的私有队列，无 future 分配 |
| `result_queue.push({t.index, std::move(res)})` | `thread_pool.cpp:198-200` | Worker 完成后结果入公共队列 |
| `try_get_result(wr)` | `thread_pool.cpp:116-123` | Aggregator 非阻塞批量 drain |
| `result_buffer[wr.index] = std::move(wr.result)` | `main.cpp` | 放入重排区待按序写出 |

> **与旧版 Future/Promise 对比：** 旧版本每帧提交需要一次 `promise` 堆分配 + 一次 `future` 原子同步 + `wait_for(0)` 轮询。新版用 `result_queue` + `mutex` 替代，Worker 批量回传结果，Aggregator 批量 drain，消除了每帧的堆分配和轮询开销。

## 5.9 线程退出顺序

```
STREAMON → ReadVideo 退出 → Aggregator 退出 → WriteVideo 退出
              │                    │                    │
          g_readFinish=true    g_processFinish=true    break
```

**原子变量传递链条：**

```
ReadVideo (采集结束)
    → g_readFinish = true
        → Aggregator 检测到 + 队列空 + tasks_inflight 空 → break
            → g_processFinish = true
                → WriteVideo 检测到 + 写队列空 → break
```

**SafeQueue 的 `stop()` 是兜底机制**（`main.cpp:649-650`），在线程 join 前调用，确保阻塞在 `dequeue` 上的线程能被唤醒退出。
