# 多线程性能量化对比

> 单线程 vs 多线程：1/3/6 Worker 下的帧率、CPU 占用、效能比实测。

---

## 测试方法

**硬件平台：** RK3588（4×A76 + 4×A55，3×NPU Core）
**测量工具：** `pidstat -t -u -p <PID> 1 10`
**测试手段：** 捕获一帧 NV12 后循环注入，绕过摄像头 30fps 限制，只测后端管线极限。

**核心指标：**
- `端到端 FPS`：WriteVideo 每 30 帧打印的真实帧率
- `进程总 CPU%`：pidstat 输出的进程级总 CPU 占用
- `效能比`：FPS ÷ 总 CPU%（每单位 CPU 算力产出的帧数）

---

## 数据对比

| Worker 数 | 帧率 | 总 CPU 占用 | 单 Worker 平均 CPU | 效能比（FPS/CPU%） | 说明 |
|-----------|------|------------|-------------------|------------------|------|
| 1 | 35fps | ~35% | 15.2% | 1.00 | 基准线 |
| 3 | **105fps** | ~96% | 15.0% | **1.09** | ✅ 线性扩展 |
| 6 | 120fps | ~111% | 10.4% | 1.08 | ❌ NPU 触顶 |

---

## 逐项分析

### 1 Worker → 35fps（基准线）

单线程串行执行：采集 → RGA → NPU 推理 → 后处理 → 编码 → 推流。

- NPU 推理期间 CPU 空闲等待（`rknn_run` 同步阻塞）
- 整条管线无任何并行，CPU 利用率仅 35%

### 3 Workers → 105fps（线性扩展）

**最重要的数据点。** 从 1→3 Worker，帧率从 35→105fps，**恰好 3 倍线性扩展**。

原因：RK3588 有 **3 个物理 NPU Core**，`i % 3` 的绑定策略让每个 Worker 独占一个 NPU Core，无任何争抢：

```
Worker 0 → NPU Core 0
Worker 1 → NPU Core 1
Worker 2 → NPU Core 2
```

总 CPU 占用 96%，说明 3 个 Worker 几乎占满了 4 个 A76 大核中的 3 个。

**效能比 1.09**——高于基准，说明 3 Worker 不仅不增加额外开销，反而因为流水线并行让 CPU 等待 NPU 的空隙被其他 Worker 利用。

### 6 Workers → 120fps（NPU 触顶）

从 3→6 Worker，帧率近乎停滞（105→120fps），且单 Worker 平均 CPU 从 15.0% **暴跌到 10.4%**——说明多出来的线程根本抢不到 NPU 资源，被底层的 `FUTEX_WAIT` 锁频繁挂起休眠，有效计算时间大幅缩水。

NPU 3 个核心已 100% 饱和，增加 Worker 只会增加锁竞争和上下文切换，不会提升吞吐。

---

## 结论

### 最佳配置：3 Workers

3 Workers 将 3 个 NPU Core 的算力完全利用，达到 **线性扩展**（效能比 1.09），是能效比和吞吐的最佳平衡点。

6 Workers 验证了**边际递减效应**——NPU 饱和后多出的线程被锁挂起，帧率几乎不涨。

### 面试话术核心

> "3 Workers 达到 **线性扩展**（105fps，效能比 1.09），验证了 3 NPU Core 的 1:1 映射是最优能效比。6 Workers 让 NPU 触顶，CPU 被 `FUTEX_WAIT` 锁频繁挂起，验证了边际递减效应。"

### 关键洞察

- **从 1→3 是硬件并行**（3 NPU Core 物理并行）
- **从 3→6 是边际递减**（NPU 饱和，多余线程被锁挂起）
