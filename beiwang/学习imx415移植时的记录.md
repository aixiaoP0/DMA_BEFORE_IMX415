**平台：** Radxa Rock 5C (RK3588S) + IMX415 (SONY STARVIS 4K)
**工作范围：** 仅涉及设备树（Device Tree）配置，不涉及内核驱动开发
**参考博客：** CSDN @weixin_49698162《RK3588 移植 IMX415 摄像头驱动完整记录》
**核心知识点：** MIPI CSI-2 协议、设备树配置、PHY 类型识别、ISP 拓扑、RKAIQ 3A 服务

## 一、IMX415 传感器硬件基础

### 1.1 核心参数

IMX415 是 SONY 的星光级（STARVIS）CMOS 图像传感器，Type 1/2.8 寸规格：

| 参数    | 值                               |
| ----- | ------------------------------- |
| 有效分辨率 | 3840×2160 (约 829 万像素，标准 4K UHD) |
| 单像素尺寸 | 1.45 µm × 1.45 µm               |
| 彩色滤镜  | Bayer RGB（拜耳阵列）                 |
| 输出位宽  | 10-bit / 12-bit RAW             |
| 最高帧率  | 4K@90fps（4-Lane 全带宽）            |
| 接口    | MIPI CSI-2 D-PHY, 4-Lane        |

### 1.2 MIPI CSI-2 物理连接

IMX415 通过 MIPI CSI-2 与 RK3588S 通信，物理连接包括：

```
IMX415                          RK3588S
┌────────┐                    ┌──────────┐
│        │  CSI_RX_D0P/N ────┤          │
│        │  CSI_RX_D1P/N ────┤  MIPI    │
│ IMX415 │  CSI_RX_D2P/N ────┤  DPHY0   │
│        │  CSI_RX_D3P/N ────┤  RX      │
│        │  CSI_RX_CLK0P/N ──┤          │
│        │                    │          │
│        │  SDA ─────────────┤ I2C3    │
│        │  SCL ─────────────┤          │
│        │                    └──────────┘
│        │  XVCLK (24MHz) ←── CRU (GPIO1_B6)
│        │  PWDN ──────────→ GPIO1_PD5
│        │  RESET ←───────── GPIO1_D2
└────────┘
```

| 信号 | 类型 | 说明 |
|------|------|------|
| `CSI_RX_D0P/N` ~ `D3P/N` | 差分数据线 ×4 | 承载 MIPI 图像数据 |
| `CSI_RX_CLK0P/N` | 差分时钟线 ×1 | D-PHY 时钟通道 |
| `SDA / SCL` | I2C | 处理器通过 I2C 写寄存器控制 Sensor |
| `XVCLK` | 时钟输入 | 外部时钟 24MHz |
| `PWDN` | GPIO 输出 | 休眠控制（Power Down） |
| `RESET` | GPIO 输出 | 复位引脚 |

### 1.3 为什么需要 4-Lane？

MIPI CSI-2 D-PHY 中，Lane 数量决定带宽上限：

```
4K@60fps RAW10 理论吞吐：
  3840 × 2160 × 10bit × 60fps ≈ 4.97 Gbps
  单 Lane 带宽：~2.5 Gbps（RK3588 DPHY 极限）
  2-Lane：5.0 Gbps → 勉强，有风险
  4-Lane：10.0 Gbps → 充足
```

所以设备树中配置 `data-lanes = <1 2 3 4>;` 使用全部 4 条 Lane 确保全带宽。

### 1.4 IMX415 的四路供电

IMX415 需要四路独立的电源轨：

| 电源域 | 电压 | 用途 |
|--------|------|------|
| AVDD（模拟） | 2.9V | 光电转换、ADC 等模拟电路，纹波直接影响画质 |
| DVDD（数字） | 1.1V | 逻辑处理、MIPI 封包等数字电路 |
| OVDD/DOVDD（接口） | 1.8V | I2C 和 MIPI 接口电平基准 |
| VDD_IO | 1.8V / 3.3V | GPIO 接口电平 |

## 二、RK3588S MIPI 接收架构

### 2.1 两种 PHY 类型

RK3588/RK3588S 内部包含两种完全不同的 MIPI 接收器：

| 类型 | 名称 | 标称速率 | 设备树前缀 | 适用场景 |
|------|------|---------|-----------|---------|
| **类型 A：纯 D-PHY** | `MIPI DPHY CSI_RX Port0` | D-PHY V1.2, 2.5Gbps | `csi2_dphy` | 低速、稳定、兼容性强 |
| **类型 B：D/C-PHY 混合** | `MIPI D/C-PHY CSI_RX Port0` | D-PHY V2.0 4.5Gbps / C-PHY V1.1 5.7Gbps | `csi2_dcphy` | 高速、新款 Combo PHY |

**如何确定用哪个？** 追踪原理图网络标号：

```
31-Pin FPC 座子 → MIPI_CSI0_RX_D0P 网络
    → 连接到原理图第 8 页的 "MIPI DPHY CSI_RX Port0"
    → 标注为 "MIPI V1.2 2.5Gbps"（纯 D-PHY）
    → 设备树中对应 csi2_dphy0
```

**验证方法：**

```bash
# 查看芯片底层定义
grep -A 5 "csi2_dphy0:" arch/arm64/boot/dts/rockchip/rk3588*.dtsi
# 输出: phy@fd5b0000, compatible = "rockchip,rk3588-csi2-dphy"

grep -A 5 "csi2_dcphy0:" arch/arm64/boot/dts/rockchip/rk3588*.dtsi
# 输出: phy@feda0000, compatible = "rockchip,rk3588-csi2-dcphy"
```

### 2.2 数据路径的四级硬件处理

MIPI 电信号进入 RK3588S 后的四级处理流水线：

```
第一级：csi2_dphy0（物理层）
  接收高速差分信号 → 串转并 → 10bit→8bit 解码
  
第二级：mipi0_csi2（协议层）
  识别 CSI-2 协议包 → CRC 校验 → 剥离协议头 → 提取像素数据
  
第三级：rkcif_mipi_lvds（视频采集桥接）
  缓冲与格式适配 → 转换为 ISP 兼容的数据流

第四级：rkisp0_vir0（图像信号处理器）
  RAW Bayer 输入 → 去马赛克 → 白平衡 → 色彩校正 → YUV 输出
```

| 级数 | 设备树节点 | 功能 |
|------|-----------|------|
| 1 | `csi2_dphy0` | 差分信号→数字信号（PHY） |
| 2 | `mipi0_csi2` | 协议解析、去包头（CSI-2） |
| 3 | `rkcif_mipi_lvds` | 缓冲与格式适配（CIF） |
| 4 | `rkisp0_vir0` | RAW→YUV 图像处理（ISP） |

### 2.3 MIPI 编码与时钟恢复原理

高速传输的核心问题：在 Mbps 级别的传输速率下，发送端和接收端如何保持同步？

**MIPI 的解决方案：**

1. **差分信号**：每路信号用 P/N 两根线传输，抗干扰强
2. **内嵌时钟**：时钟信息通过 8B/10B 编码嵌入数据流中
3. **8B/10B 编码**：
   - 8-bit 原始数据 → 查表编码为 10-bit
   - 保证足够多的 0↔1 跳变（时钟恢复必要条件）
   - 维持 DC 平衡（防止基线漂移）
4. **PLL 时钟恢复**：
   ```
   发送端：24MHz 基准 → PLL 倍频 → 高速串行发送
   接收端：解析数据流跳变 → PLL 锁定频率 → 生成接收时钟
   ```

这种设计的优势：**无需专门的时钟同步线，每根数据线独立工作**。

## 三、设备树配置（核心章节）

### 3.1 文件选择策略

**参考文件：** `rk3588-evb1-imx415.dtsi` 或 `rk3588-evb2-imx415.dtsi`

**为什么有这么多 EVB 文件？** Rockchip SDK 的 EVB（Evaluation Board）对应不同硬件形态：

| EVB 变体 | 对应形态 |
|----------|---------|
| EVB1 | 全功能旗舰大板 |
| EVB2/3/7 | IPC、平板等特定领域精简板 |
| 后缀 lp4/lp5 | 内存类型差异 |

这些文件对同一颗 Sensor 的内部节点配置几乎一样，差异仅在于 I2C 总线和 MIPI PHY 通道。

**建议做法（模块化分离法）：**

1. 创建独立设备树文件 `rk3588s-rock-5c-imx415.dtsi`
2. 将摄像头相关拓扑全部写在该文件中
3. 在主设备树 `rk3588s-rock-5c.dts` 末尾引用：

```devicetree
#include "rk3588s-rock-5c-imx415.dtsi"
```

### 3.2 Sensor 基础节点

IMX415 的 I2C 地址为 `0x1a`（7-bit 地址，即 `0x34` 的右移一位），挂载在 I2C3 总线上：

```devicetree
&i2c3 {
    status = "okay";
    pinctrl-names = "default";
    pinctrl-0 = <&i2c3m0_xfer>;

    imx415: imx415@1a {
        compatible = "sony,imx415";
        reg = <0x1a>;

        /* 时钟：24MHz 来自 CRU 的 MIPI_CAMERA1_CLK_M0 */
        clocks = <&cru CLK_MIPI_CAMARAOUT_M0>;
        clock-names = "xvclk";

        /* 引脚复用：GPIO1_B6 配置为 camera1_clk 功能 */
        pinctrl-names = "default";
        pinctrl-0 = <&mipim0_camera1_clk>;

        /* 电源域：RK3588_PD_VI */
        power-domains = <&power RK3588_PD_VI>;

        /* 休眠引脚：GPIO1_PD5，高电平有效 */
        pwdn-gpios = <&gpio1 RK_PD5 GPIO_ACTIVE_HIGH>;

        /* 复位引脚：GPIO1_D2 */
        reset-gpios = <&gpio1 RK_PD2 GPIO_ACTIVE_LOW>;

        /* 模组元数据（详见 3.6 节） */
        rockchip,camera-module-index = <0>;
        rockchip,camera-module-facing = "back";
        rockchip,camera-module-name = "RADXA-CAMERA-4K";
        rockchip,camera-module-lens-name = "DEFAULT";

        /* 数据输出端口 */
        port {
            imx415_out0: endpoint {
                remote-endpoint = <&mipi_in_ucam0>;
                data-lanes = <1 2 3 4>;
            };
        };
    };
};
```

### 3.3 时钟配置详解

**物理路径：**

```
CRU（RK3588 时钟管理单元）
  → MIPI_CAMERA1_CLK_M0（GPIO1_B6 复用功能 2）
  → 电平转换电路（3.3V → 1.8V）
  → IMX415 XVCLK 引脚（24MHz）
```

**设备树中的时钟宏：**

```devicetree
clocks = <&cru CLK_MIPI_CAMARAOUT_M0>;
```

> **⚠ 注意 Rockchip 的祖传拼写错误：** `CAMERA` 被误写为 `CAMARA`（少了个 E）。这个拼写错误因为向前兼容被永久保留在了 BSP 内核中。

**引脚复用配置：**

```devicetree
/* 在 rk3588s-pinctrl.dtsi 中的定义 */
mipim0_camera1_clk: mipim0-camera1-clk {
    rockchip,pins =
        <1 RK_PB6 2 &pcfg_pull_none>;
};
```

**为什么有两个 CLK？（CAMERA1_CLK 和 CAMERA2_CLK）** 这个 31-Pin 座子设计支持"单接口双摄像头"模组，两颗 Sensor 需要各自独立的 24MHz 时钟源。IMX415 是单摄 4-Lane，只用 CAMERA1_CLK。

### 3.4 电平转换电路（3.3V → 1.8V）

原理图中存在基于 N 沟道 MOSFET（2SK3018）的电平转换电路：

```
主控 3.3V                   摄像头 1.8V
MIPI_CAMERAx_CLK_M0 ──┬── R(1.8K) ── VCC_3V3
                      │
                    Drain
                      ↑
            2SK3018 (N-MOSFET)
                      ↓
                    Source
                      │
                    R(10K)
                      │
MIPI_CAMERAx_CLK_1 ──┴── VCC_1V8
```

G = 1.8V（通过 R=1K 上拉）

**工作原理：**

| 输入电平 | MOSFET 状态 | 输出电平 |
|----------|------------|---------|
| 3.3V（高） | 关断 | 1.8V（被 R=10K 上拉） |
| 0V（低） | 导通 | 0V（被拉到 GND） |

### 3.5 四级 ISP 拓扑连接（port/endpoint）

MIPI 信号到 ISP 的设备树描述使用 **port/endpoint** 四级串联。每级的 endpoint 建立"数据流水线的单向前进"关系，并双向引用确保连接完整性：

```devicetree
/* ========== 第一级：PHY 层 ========== */
&csi2_dphy0 {
    status = "okay";
    ports {
        #address-cells = <1>;
        #size-cells = <0>;

        /* 输入端口：接 Sensor */
        port@0 {
            reg = <0>;
            #address-cells = <1>;
            #size-cells = <0>;

            mipi_in_ucam0: endpoint@1 {
                reg = <1>;
                remote-endpoint = <&imx415_out0>;
                data-lanes = <1 2 3 4>;
            };
        };

        /* 输出端口：接 CSI-2 协议层 */
        port@1 {
            reg = <1>;
            #address-cells = <1>;
            #size-cells = <0>;

            csidcphy0_out: endpoint@0 {
                reg = <0>;
                remote-endpoint = <&mipi0_csi2_input>;
            };
        };
    };
};

/* ========== 第二级：CSI-2 协议层 ========== */
&mipi0_csi2 {
    status = "okay";
    ports {
        #address-cells = <1>;
        #size-cells = <0>;

        port@0 {
            reg = <0>;
            #address-cells = <1>;
            #size-cells = <0>;

            mipi0_csi2_input: endpoint@1 {
                reg = <1>;
                remote-endpoint = <&csidcphy0_out>;
            };
        };

        port@1 {
            reg = <1>;
            #address-cells = <1>;
            #size-cells = <0>;

            mipi0_csi2_output: endpoint@0 {
                reg = <0>;
                remote-endpoint = <&cif_mipi_in0>;
            };
        };
    };
};

/* ========== 第三级：CIF 桥接 ========== */
&rkcif_mipi_lvds {
    status = "okay";
    port {
        cif_mipi_in0: endpoint {
            remote-endpoint = <&mipi0_csi2_output>;
        };
    };
};

&rkcif_mipi_lvds_sditf {
    status = "okay";
    port {
        mipi_lvds_sditf: endpoint {
            remote-endpoint = <&isp0_vir0>;
        };
    };
};

/* ========== 第四级：ISP ========== */
&rkisp0_vir0 {
    status = "okay";
    port {
        #address-cells = <1>;
        #size-cells = <0>;

        isp0_vir0: endpoint@0 {
            reg = <0>;
            remote-endpoint = <&mipi_lvds_sditf>;
        };
    };
};
```

**关键理解：port/endpoint 的实质**

| 术语 | 物理类比 | 含义 |
|------|---------|------|
| `port` | 硬件模块的数据接口 | 像 USB 接口 |
| `endpoint` | 接口上的具体连接点 | 像插在 USB 上的线端 |
| `remote-endpoint` | 数据流向的下一站 | 指向对端 endpoint |
| `data-lanes` | 使用的物理 Lane 编号 | `<1 2 3 4>` 表示全部 4 条 Lane |

### 3.6 模组元数据与 IQ 文件匹配

```devicetree
rockchip,camera-module-index = <0>;      // 摄像头编号
rockchip,camera-module-facing = "back";   // 后置
rockchip,camera-module-name = "RADXA-CAMERA-4K";
rockchip,camera-module-lens-name = "DEFAULT";
```

**IQ 文件的作用：** ISP 在初始化时会去 `/etc/iqfiles/` 目录寻找对应调优文件。文件名拼接规则为 `[sensor]_[name]_[lens-name].json`。

Radxa 5C 系统中实际存在的文件：

```
/etc/iqfiles/imx415_CMK-OT2022-PX1_IR0147-50IRC-8M-F20.json      (原厂 EVB 文件)
/etc/iqfiles/imx415_RADXA-CAMERA-4K_DEFAULT.json                  (Radxa 官方调优)
```

**选择策略：** 优先匹配 Radxa 官方的 `RADXA-CAMERA-4K` 调优文件，因为 Radxa 工程师专门为 IMX415 + Radxa 5C 做过色彩和白平衡调优。

验证方法：

```bash
ls -l /etc/iqfiles/ | grep -i "imx415"
```

### 3.7 I2C 引脚复用

```devicetree
&i2c3 {
    pinctrl-names = "default";
    pinctrl-0 = <&i2c3m0_xfer>;
};
```

Pinctrl 子系统的角色：RK3588S 的 GPIO 引脚具有多重功能（引脚复用）。`i2c3m0_xfer` 将 GPIO1_C0 和 GPIO1_C1 从普通 GPIO 切换为 I2C3 的 SDA/SCL 功能。

但 Radxa 5C 的原理图中，I2C3_SDA_M0 是 GPIO1_C0 的默认功能，所以即使不配 pinctrl，I2C 也可能工作。但为了代码清晰和跨平台兼容，建议保留。

```devicetree
/* 在 rk3588-pinctrl.dtsi 中的定义 */
i2c3m0_xfer: i2c3m0-xfer {
    rockchip,pins =
        <1 RK_PC1 9 &pcfg_pull_none_smt>,  /* i2c3_scl_m0 */
        <1 RK_PC0 9 &pcfg_pull_none_smt>;  /* i2c3_sda_m0 */
};
```

### 3.8 电源门控（Power Gating）

```devicetree
power-domains = <&power RK3588_PD_VI>;
```

**RK3588_PD_VI（Video Input 电源域）**控制 SoC 内部 MIPI 控制器、ISP 等视频输入相关模块的供电。不开启这个域，MIPI 控制器和 ISP 无法工作。

**物理原理：**

- SoC 内部划分多个电源域（Power Domain）
- 每个域入口处设置 MOS 晶体管开关
- 关断开关 → 物理切断电源 → 漏电流降为 0
- PMU（电源管理单元）控制这些开关

### 3.9 删除的节点及原因

从参考设备树中删除了以下不兼容的属性：

| 删除的属性 | 原因 |
|-----------|------|
| `lens-focus = <&cam_ircut0>` | Radxa 5C 的 IMX415 模组是定焦模组，无 VCM 马达，保留此字段会导致 Probe 失败 |
| `avdd-supply = <&vcc_mipidcphy0>` | Radxa 5C 摄像头供电由主板电源开关直接控制，`vcc_mipidcphy0` 节点不存在 |

## 四、调试与验证

### 4.1 内核编译部署

```bash
# 在开发主机上编译
make ARCH=arm64 rockchip_linux_defconfig
make ARCH=arm64 rk3588s-rock-5c.dtb

# 将 DTB 推送到板子，替换 /boot/dtb/ 下的文件
scp arch/arm64/boot/dts/rockchip/rk3588s-rock-5c.dtb root@192.168.x.x:/boot/dtb/

# 重启
reboot
```

### 4.2 I2C 握手验证

重启后，检查 Sensor 是否成功 Probe：

```bash
dmesg | grep -i imx415
# 期望输出：imx415 3-001a: Detected imx415 id e0
```

常见失败原因：

| 失败原因 | 排查方向 |
|----------|---------|
| I2C 地址错误 | IMX415 是 0x1a（不是 0x34） |
| 复位引脚电平不对 | 检查 reset-gpios 电平极性 |
| 24MHz 时钟未起振 | 检查 CRU 时钟配置和引脚复用 |
| 电源域未开启 | 检查 power-domains 是否正确 |

### 4.3 ISP 节点确认

```bash
# 查看多媒体管道拓扑
media-ctl -d /dev/media0 -p | grep "rkisp_mainpath"
# 期望输出：device node name /dev/video11
```

ISP 输出的节点号可能是 `/dev/video11`，需在 `main.cpp` 中修改对应的 `open()` 路径。

### 4.4 RKAIQ 3A 服务

> **绝对不可省略的一步！** 不启动 3A 服务，画面会全黑或严重偏色。

```bash
# 在后台启动 IMX415 的 3A 算法
rkaiq_3A_server -d /dev/video11 &

# 验证加载了正确的 IQ 文件
dmesg | grep -i "iqfile"
# 期望输出：成功加载 imx415_RADXA-CAMERA-4K_DEFAULT.json
```

## 五、关键知识点速查

### 5.1 常见陷阱

| # | 陷阱 | 说明 |
|---|------|------|
| 1 | Rockchip "CAMARA" 拼写错误 | `CLK_MIPI_CAMARAOUT_M0` 而非 `CAMERA` |
| 2 | I2C 地址 | 7-bit 地址 0x1a（不是 0x34） |
| 3 | PHY 类型区分 | `csi2_dphy`（纯 D-PHY）vs `csi2_dcphy`（Combo PHY） |
| 4 | 电平转换 | 主控 3.3V → 摄像头 1.8V，MOSFET 实现 |
| 5 | 电源域 | 必须开启 `RK3588_PD_VI` |
| 6 | IQ 文件匹配 | 文件名拼接规则 `[sensor]_[name]_[lens-name].json` |
| 7 | 带电插拔 | MIPI 接口严禁带电插拔，极易烧毁 D-PHY |
| 8 | RKAIQ 服务 | 每次启动应用前必须启动 `rkaiq_3A_server` |
