## 2. `new/delete` vs `malloc/free`

**对应八股.pdf 页码**：第 28 页
**PDF 核心考点**：malloc 分配堆内存不调用构造函数；new 分配+构造；free 不调用析构函数

**代码对照** — 新架构相比旧架构有重大变化：

| 场景 | 旧架构 | 新架构 | 变化 |
|------|--------|--------|------|
| NV12 拷贝 | ReadVideo: `malloc+memcpy` (Path B) | **已移除** (Path C RGA 直写 NPU) | ✅ 零拷贝 |
| Worker 输出 NV12 | `malloc(nv12_size)` | `malloc(nv12_size)` | 不变（编码器输入） |
| YOLO 模型实例 | `make_shared<Yolov5s>()` | `make_shared<Yolov5s>()` | 不变 |
| WorkerContext 数组 | `vector<WorkerContext>` | `unique_ptr<WorkerContext[]>` | ✅ 更安全的 RAII |

**关键差异：**

**新架构 ReadVideo 中不再有 malloc+memcpy NV12**（`main.cpp:249-274`）：
```cpp
// 新架构 ReadVideo：Path C 用 RGA 硬件 DMA 直写 NPU，无 CPU 拷贝
rga_buffer_handle_t src_handle_c = importbuffer_fd(buffer_info.dmabuf_fd, nv12_size);
rga_buffer_handle_t dst_handle_c = importbuffer_fd(npu_fd, 640 * 640 * 3);
imresize(src_c, dst_c);  // 硬件完成：缩放 + NV12→RGB + 写入 NPU
// 没有 malloc！没有 memcpy！
```

Worker 中唯一剩下的 malloc（`thread_pool.cpp:178`）——编码器输入用：
```cpp
uint8_t* pixel_buffer = (uint8_t*)malloc(nv12_size);
// nv12_size = width × height × 3 / 2 ≈ 1.38MB @ 720p
```

> **面试价值：** 新架构从 "2 次 malloc + 2 次 memcpy"（ReadVideo + Worker）降到 "1 次 malloc"（仅 Worker 输出给编码器）。这是零拷贝重构的核心成果。面试中可以从这个角度展示你对 C++ 内存管理与嵌入式优化的综合理解。

## 6. C++ `map` / `set`（STL 关联容器）

**对应八股.pdf 页码**：第 53-54 页
**PDF 核心考点**：map/set 底层 RB-Tree、map `[]` vs `find`、set 自动去重

**代码对照** — 新架构有重大变化：

**新架构 Aggregator** — `main.cpp:325`：
```cpp
map<int, ProcessResult> result_buffer;  // key=帧序号, value=已完成结果
```

旧架构用的是 `map<int, future<ProcessResult>> tasks_inflight`。新架构移除了 `future`，直接存储 `ProcessResult`：

```cpp
// 步骤A：提交后不存 future，结果通过 result_queue 回传
gthreadpool.submit_to_worker(index % 4, index, frame, width, height);
// 无返回值！不再存入 map

// 步骤B：try_get_result 批量 drain 结果
WorkerResult wr;
while(gthreadpool.try_get_result(wr)) {
    result_buffer[wr.index] = std::move(wr.result);  // 结果存入 map
}

// 步骤C：find(nextWriteIndex) 按序写出
auto it = result_buffer.find(nextWriteIndex);
while(it != result_buffer.end()) {
    g_SafeQueueWrite.enqueue(outputFD);
    result_buffer.erase(it);
    nextWriteIndex++;
    it = result_buffer.find(nextWriteIndex);
}
```

`post_process.cpp` 中 NMS 的 `set` 去重使用不变：
```cpp
std::set<int> class_set(begin(classId), end(classId));
for (const int& id : class_set) {
    nms(vC, detect_boxes, classId, indexArray, id, nms_threshold);
}
```

> **面试对比：** 旧架构用 `map<int, future>` 是因为 `future` 在提交时就被创建（通过 `promise.get_future()`），map 存储的是"异步结果凭证"。新架构用 `map<int, ProcessResult>` 存储的是"已经完成、等待按序写出的结果"，更直观，也减少了每帧一个 promise 的堆分配开销。
>
> 两者都用 map 而不是 unordered_map 的原因相同：需要 `find(nextWriteIndex)` 的**按 key 顺序遍历**语义。

## 10. ~~`std::future` / `std::promise`~~ → 已移除

> ⚠️ **新架构已完全移除 future/promise 机制。**
>
> - 旧架构：`Task` 结构体包含 `std::promise<ProcessResult>`，`submit_task_async` 返回 `std::future<ProcessResult>`，Worker 完成后 `promise.set_value()`，Aggregator 通过 `future.wait_for(0)` 轮询
> - 新架构：`submit_to_worker` **无返回值**，Worker 完成后结果入 `result_queue`，Aggregator 通过 `try_get_result()` 批量 drain

**新架构的实现（`thread_pool.cpp:82-99`）：**
```cpp
void ThreadPool::submit_to_worker(int worker_id, int index, const cv::Mat& img,
                                   int width, int height)
{
    Task t;
    t.index = index;
    t.img = img;       // cv::Mat 浅拷贝（引用计数）
    t.width = width;
    t.height = height;

    tasks_in_flight++;  // 原子递增

    // 入队到指定 Worker 的私有队列
    lock_guard<mutex> lock(worker_ctx[worker_id].mtx);
    worker_ctx[worker_id].tasks.push(std::move(t));
    // 锁释放
    worker_ctx[worker_id].cv.notify_one();  // 通知 Worker
    // 无 future 返回！
}
```

**Task 结构体简化（`thread_pool.h:40-45`）：**
```cpp
typedef struct Task {
    int index = 0;       // 帧序号
    cv::Mat img;         // BGR图像（画框用）
    int width = 0;       // 原始宽度
    int height = 0;      // 原始高度
} Task;
// 旧架构 Task 中的 std::promise<ProcessResult>、uint8_t* nv12_data 已移除
```

> **面试价值：** 这是新架构最核心的重构。可以这么跟面试官解释：
>
> "旧架构用 `future/promise` 是因为每个任务需要异步回传结果。但每帧都要做一次 `promise` 堆分配+ `future` 原子同步 + `wait_for` 轮询，在 30fps 下开销可观。新架构改为 `result_queue` + `try_get_result`：Worker 完成后把结果 push 到公共队列，Aggregator 非阻塞批量 drain。这比 future 更轻量——从'每个任务一个异步通道'变成了'所有任务共享一个结果池'。"

## 14. 条件变量 `condition_variable`（新架构中三处使用）

**对应八股.pdf 页码**：【高频面试考点，PDF 25-66 页未直接覆盖】
**经典八股考点**：虚假唤醒、predicate wait、unique_lock、notify_one vs notify_all

**新架构有三处条件变量使用：**

### 14.1 SafeQueue 双条件变量（与旧架构一致）

`SafeQueue.h:21-44`：
```cpp
void enqueue(const T& t) {
    unique_lock<mutex> lock(m);
    cond_not_full.wait(lock, [this]{return q.size() < maxSize || stop_flag;});
    q.push(t);
    cond_not_empty.notify_all();
}

bool dequeue(T& t) {
    unique_lock<mutex> lock(m);
    cond_not_empty.wait(lock, [this]{return !q.empty() || stop_flag;});
    if (stop_flag && q.empty()) return false;
    t = q.front(); q.pop();
    cond_not_full.notify_all();
    return true;
}
```

| 八股考点 | 代码实现 |
|---------|---------|
| 虚假唤醒 | `wait()` 的 predicate 参数自动处理 |
| 双条件变量 | `cond_not_full` + `cond_not_empty`，避免生产者唤醒生产者 |
| Lambda predicate | `[this]{return q.size() < maxSize;}` — C++11 Lambda |

### 14.2 Per-Worker 条件变量（新架构新增）

`thread_pool.cpp:156-158` — Worker 等待任务：

SafeQueue.h:
```cpp
unique_lock<mutex> lock(worker_ctx[id].mtx);
worker_ctx[id].cv.wait(lock, [this, id] {
    return !worker_ctx[id].tasks.empty() || !run_flag;
});
```

`thread_pool.cpp:98` — Aggregator 通知 Worker：
```cpp
worker_ctx[worker_id].cv.notify_one();  // 只唤醒目标 Worker
```

**为什么这里用 `notify_one` 而不是 `notify_all`？**
- 每 Worker 独立队列，每个任务只发送给一个确定的 Worker
- 如果用 `notify_all`，所有 Worker 都会被唤醒，但只有 target Worker 队列非空，其他 Worker 的 predicate 返回 false 继续 wait——这是无效唤醒，浪费 CPU

### 14.3 对比：SafeQueue 中用 `notify_all` vs Worker 中用 `notify_one`

| 位置 | 使用 | 原因 |
|------|------|------|
| SafeQueue `enqueue` | `cond_not_empty.notify_all()` | 可能有多个消费者等待（虽然实际只有 Aggregator 一个消费者，但设计上支持多消费者） |
| SafeQueue `dequeue` | `cond_not_full.notify_all()` | 可能有多个生产者等待（ReadVideo 一个即可，但设计通用） |
| `submit_to_worker` | `cv.notify_one()` | 明确的单消费者（目标 Worker） |

## 19. `unique_ptr` 数组管理（新架构新增）

> ⚠️ **新架构独有，旧架构中使用 `vector`**

**代码：** `thread_pool.h:115`：
```cpp
std::unique_ptr<WorkerContext[]> worker_ctx;
```

`WorkerContext` 包含 `mutex` 和 `condition_variable`，两者都是**不可拷贝**（non-copyable）的。`unique_ptr<WorkerContext[]>` 通过 `new[]` 在堆上构造数组，完美解决这个问题：

```cpp
// thread_pool.cpp:57-58
worker_ctx.reset(new WorkerContext[num_threads]);
// 等价于 new WorkerContext[num_threads]{WorkerContext{}, WorkerContext{}, ...}
```

**为什么不用 `vector<WorkerContext>`？**
- `vector` 的默认构造会调用 `WorkerContext()`，这没问题
- 但 `vector` 的某些操作（如 `reserve` 触发的 reallocation）需要移动/拷贝元素，而 `mutex` 和 `condition_variable` 不可拷贝也不可移动
- `unique_ptr<WorkerContext[]>` 不存在 reallocation，是安全的

**为什么不用 `vector<unique_ptr<WorkerContext>>`？**
- 可以，但多了一层间接性和额外的堆分配
- `unique_ptr<WorkerContext[]>` 是连续内存布局（cache-friendly），一次分配

> **面试价值：** 这是一个非常经典的 C++ 问题：如何管理一组包含不可拷贝成员的数组。回答时能展示你对 `unique_ptr` 数组特化（`T[]` vs `T`）的理解，以及对 `mutex` 不可拷贝特性的 awareness。

> **面试追问：** 为什么用 `atomic` 而不是 `mutex`？→ 这些标志只做三件事：读、写 true、写 false。没有"读-改-写"复合操作，用 `atomic` 就够了。`atomic<bool>` 在 x86 上编译为单条 `MOV` 指令（~1ns），而 `mutex` 涉及用户态 futex 或系统调用（~25-200ns）。对 `is_busy` 这种每帧（33ms）要被 ReadVideo 检查一次的变量，用 `atomic` 可以避免不必要的锁开销。

## 21. Lambda 表达式（C++11）

**代码对照** — 新架构 SafeQueue 中使用 Lambda 作为条件变量 predicate：

`SafeQueue.h:25`：
```cpp
cond_not_full.wait(lock, [this]{ return q.size() < maxSize; });
//                ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
//                捕获 this 指针，返回 bool 的 Lambda
```

`SafeQueue.h:35`：
```cpp
cond_not_empty.wait(lock, [this]{ return !q.empty(); });
```

`thread_pool.cpp:157`：
```cpp
worker_ctx[id].cv.wait(lock, [this, id] {
    return !worker_ctx[id].tasks.empty() || !run_flag;
});
// 捕获 this 和 id 两个变量
```

**为什么不直接用函数？** Lambda 可以直接捕获 `this` 和局部变量 `id`，等价于一个"内联函数对象"。如果用普通函数或函数指针，需要额外传递上下文参数，或用 `bind`——Lambda 更简洁，代码生成质量也更好（编译器更容易内联）。

> **面试价值：** 条件变量的 predicate 是 Lambda 在嵌入式 C++ 中的典型应用场景。注意这里用的 `[this]` 按引用捕获 SafeQueue 对象，Lambda 内直接访问成员变量 `q.size()`——如果按值捕获 `[this]`，效果相同（捕获指针的副本），但捕获 `[=]` 会按值捕获所有自动变量，对于 `queue` 这样的容器不合适。
