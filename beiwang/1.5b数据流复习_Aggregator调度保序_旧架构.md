## 5.1 Aggregator 要解决什么问题

线程池中有 4 个 Worker 并行推理，每个 Worker 处理速度不同：

```
ReadVideo 采集顺序:  frame0  frame1  frame2  frame3  frame4  frame5 ...
                        │       │       │       │       │       │
                        ▼       ▼       ▼       ▼       ▼       ▼
                        W0      W1      W2      W3      W0      W1   (round-robin)
                    完成慢    完成快    完成中    完成快    完成慢      ...
                        │       │       │       │       │       │
                        ▼       ▼       ▼       ▼       ▼       ▼
                    结果顺序:  frame0  frame1  frame2  frame3  frame4  ...
```

如果没有保序机制，推理完成的帧会乱序到达 WriteVideo 线程，导致视频画面帧序跳跃。旧架构通过 **`future` + `map` + `nextWriteIndex`** 的组合实现乱序执行但顺序提交。

这个问题的本质是：**并行提交 + 串行输出之间的阻抗不匹配**。

---

## 5.2 核心机制：future/promise 异步回传

旧架构中，Aggregator 不直接与 Worker 通信。它通过 `std::promise` / `std::future` 建立了一条异步结果通道：

```
Aggregator 提交:
    auto future = gthreadpool.submit_task_async(index, img, nv12_data, ...);
    tasks_inflight[index] = std::move(future);
    
    若干时间后...
    
    auto status = tasks_inflight[nextWriteIndex].wait_for(0ms);
    if (status == future_status::ready) {
        ProcessResult result = tasks_inflight[nextWriteIndex].get();
        // 写入输出队列
    }
```

| 角色 | 类型 | 职责 |
|------|------|------|
| `std::promise<ProcessResult>` | 生产者端 | Worker 推理完成后调用 `set_value(res)` 填充结果 |
| `std::future<ProcessResult>` | 消费者端 | Aggregator 持有，通过 `wait_for`/`get` 获取结果 |
| `map<int, future<ProcessResult>> tasks_inflight` | 索引容器 | 帧序号 → future 的映射 |

**提交调用链：**

```
aggregatorThreadFunc
    └→ gthreadpool.submit_task_async(index, frame, nv12_data, ...)  // thread_pool.h:84
          └→ Task t.promise.get_future() → 返回 future              // thread_pool.cpp:96
          └→ tasks.push(move(t))                                    // thread_pool.cpp:108
          └→ task_cond.notify_one()                                 // thread_pool.cpp:110
                └→ Worker 线程被唤醒 → processing...

Worker 完成后:
    └→ t.promise.set_value(res)                                     // thread_pool.cpp:237
          └→ future 变为 ready 状态
          └→ Aggregator 的 wait_for(0) 返回 future_status::ready
```

## 5.3 数据结构：`map<int, future<ProcessResult>> tasks_inflight`

```cpp
map<int, future<ProcessResult>> tasks_inflight;
int nextWriteIndex = 0;
```

| 变量 | 类型 | 作用 |
|------|------|------|
| `tasks_inflight` | `map<int, future<ProcessResult>>` | key=帧序号, value=该帧推理结果的 future |
| `nextWriteIndex` | `int` | 期望下一个写入输出队列的帧序号，单调递增 |

`tasks_inflight` 中同时存在多个 future，每个 future 对应一帧正在推理中的任务。Aggregator 通过 `wait_for(0)` 非阻塞轮询检查当前应写帧是否已完成。

## 5.4 主循环：三个阶段

### 步骤 A：提交循环（第 316-328 行）

```cpp
while (!g_SafeQueueRead.empty() && tasks_inflight.size() < 20) {
    if (g_SafeQueueRead.dequeue(inputFD)) {
        auto future = gthreadpool.submit_task_async(
            inputFD.index, inputFD.frame,
            inputFD.nv12_data, inputFD.data_size,
            inputFD.width, inputFD.height);
        tasks_inflight[inputFD.index] = std::move(future);
    }
    is_idle = false;
}
```

关键行为：

- **限流**：`tasks_inflight.size() < 20` — 最多 20 帧在途，防止内存溢出
- **异步提交**：`submit_task_async` 将任务放入全局 `SharedQueue<Task>` 后立即返回 future
- **无条件提交**：旧架构没有 `is_busy` 丢帧机制，所有采集到的帧都提交推理

### 步骤 B：收集循环（第 332-364 行）

```cpp
auto it = tasks_inflight.find(nextWriteIndex);
while (it != tasks_inflight.end()) {
    auto status = it->second.wait_for(std::chrono::milliseconds(0));
    if (status == std::future_status::ready) {
        ProcessResult result = it->second.get();

        FrameData outputFD;
        outputFD.index = nextWriteIndex;
        outputFD.frame = result.processed_img.clone();
        outputFD.nv12_data = result.nv12_data;
        outputFD.data_size = result.data_size;

        g_SafeQueueWrite.enqueue(outputFD);
        tasks_inflight.erase(it);
        nextWriteIndex++;
        it = tasks_inflight.find(nextWriteIndex);
        is_idle = false;
    } else {
        break;
    }
}
```

关键行为：

- **`wait_for(0)` 非阻塞轮询** — 不阻塞当前线程，仅检查 future 是否 ready
- **`future.get()` 获取结果** — 仅在 `wait_for` 确认 ready 后调用，不会阻塞
- **顺序写入** — 只检查 `nextWriteIndex` 对应的帧，即使 frame 5 已经完成而 frame 3 未完成，frame 5 也必须等待
- **连续写入** — 用 `while` 而非 `if`：frame 3 完成写入后 `nextWriteIndex=4`，如果 frame 4 也已 ready，继续处理

### 步骤 C：退出检测（第 367-371 行）

```cpp
if (g_readFinish && g_SafeQueueRead.empty() && tasks_inflight.empty()) {
    cout << "处理线程已结束" << endl;
    break;
}
```

退出条件三要素**缺一不可**：

| 条件 | 说明 |
|------|------|
| `g_readFinish == true` | ReadVideo 已停止采集 |
| `g_SafeQueueRead.empty()` | 读队列中的残留帧已全部取出 |
| `tasks_inflight.empty()` | 所有已提交的推理任务都已用 `get()` 取出结果并 `erase` |

## 5.5 Head-of-Line Blocking（队头阻塞）

这是保序机制的固有代价。假设推理耗时分布如下：

```
提交顺序:  frame0  frame1  frame2  frame3  frame4  frame5  frame6  frame7
            │       │       │       │       │       │       │       │
            ▼       ▼       ▼       ▼       ▼       ▼       ▼       ▼
           W0      W1      W2      W3      W0      W1      W2      W3
          推理慢   推理快   推理快   推理快   推理快   推理快   推理快   推理快
                                                                    
完成时刻:  t+80   t+10   t+15   t+12   t+85   t+20   t+22   t+18
           ↑                  ↑                              ↑
       frame0 最慢      但 frame0 不完成，                   frame4 虽然完成了
       frame1~3 已就绪  nextWriteIndex 无法推进             但必须等 frame0 先写入
```

**时序模拟：**

```
t+0:   提交 frame0~3 (limit=20, 全部入队)
t+10:  frame1 完成 → wait_for(0) 检查 nextWriteIndex=0 → 未就绪 → 跳过
t+12:  frame3 完成 → 同上
t+15:  frame2 完成 → 同上
       (frame0 阻塞了所有后面的帧)
t+80:  frame0 完成 → wait_for(0) 找到 nextWriteIndex=0 → ready!
       写入 frame0, nextWriteIndex=1
       写入 frame1, nextWriteIndex=2
       写入 frame2, nextWriteIndex=3
       写入 frame3, nextWriteIndex=4
       frame4 还在推理中，nextWriteIndex=4 尚未 ready → 等待
```

**队头阻塞期间系统在做什么？**

检查 `is_idle`：

```cpp
// 步骤 A：提交循环 — 可能还在提交新帧（如果读队列非空且未达上限）
// 步骤 B：nextWriteIndex 未就绪 → wait_for 返回 timeout → break
// 都没有 → is_idle = true
if (is_idle) {
    std::this_thread::sleep_for(std::chrono::milliseconds(1));
}
```

在 frame0 阻塞期间，如果读队列还有帧且 `tasks_inflight.size() < 20`，Aggregator 会继续提交新帧（步骤 A 可以执行）。只有步骤 A 和步骤 B 都无事可做时才休眠。

> **工程权衡：** 旧架构没有主动丢帧机制。如果某一帧推理时间异常长（>100ms），后续所有已完成帧都被阻塞，造成输出帧率陡降。这是"保序"与"吞吐"之间不可调和的矛盾——要么接受队头阻塞保证时序正确，要么设计丢帧机制牺牲某些帧保证整体流畅度。

## 5.6 内存所有权链

旧架构中 NV12 数据的生命周期：

```
ReadVideo:  malloc(nv12) → memcpy(NV12 from DMABUF)
                │
                ▼  FrameData.nv12_data 传入 Aggregator
Aggregator: submit_task_async(index, ..., nv12_data, nv12_size, ...)
                │  Task.nv12_data = nv12_data (指针传递，不拷贝)
                ▼
SharedQueue<Task> → Worker 线程出队
                │
                ▼
Worker:  使用 nv12_data 完成推理后 → free(t.nv12_data)
                │
         BGR→NV12 → malloc → ProcessResult.nv12_data
                │
                ▼  result = t.promise.set_value(res) → future.get() 返回
Aggregator: outputFD.nv12_data = result.nv12_data (指针传递)
                │
                ▼  g_SafeQueueWrite.enqueue(outputFD)
WriteVideo: process_frame(output_FD.nv12_data) → 编码推流 → free(output_FD.nv12_data)
```

**关键点：** 每次帧数据的 NV12 缓冲区只被释放一次，但经过了三次指针传递：

| 阶段 | 分配/释放 | 操作 |
|------|----------|------|
| ReadVideo | `malloc` | 分配 NV12 拷贝 |
| Worker | `free` | **释放输入 NV12**（推理用完后） |
| Worker | `malloc` | **新分配输出 NV12**（BGR→NV12 转换结果） |
| WriteVideo | `free` | **释放输出 NV12**（编码推流用完后） |

每一帧涉及 **两次 malloc + 两次 free**，分别对应推理输入和编码输出。

## 5.7 空闲检测与 CPU 节流

```cpp
if (is_idle) {
    std::this_thread::sleep_for(std::chrono::milliseconds(1));
}
```

- `is_idle` 初始化为 `true`，只有步骤 A（提交新帧）或步骤 B（收集结果）有实际工作时才设为 `false`
- 既没有新帧可提交、也没有已完成的下一帧可收集时，线程休眠 1ms 避免空转
- 休眠时间短（1ms），不会引入明显延迟

## 5.8 完整数据流

```
ReadVideo                  Aggregator                         Worker[i]
    │                          │                                  │
    │  enqueue ────────► g_SafeQueueRead                           │
    │                          │                                  │
    │                     dequeue                                   │
    │                     submit_task_async(index, ...) ──────►    │
    │                          │                      全局任务队列  │
    │                     tasks_inflight[index] =      task_cond   │
    │                       std::move(future)        notify_one    │
    │                          │                                  │
    │                     wait_for(0) ◄─ wait_for(0) ◄─── promise  │
    │                     (轮询 nextWriteIndex)       set_value    │
    │                          │                                  │
    │                     future.get() → ProcessResult             │
    │                          │                                  │
    │                     enqueue ──────► g_SafeQueueWrite          │
    │                          │                                  │
    │               WriteVideo │                                  │
    │                  dequeue ◄┘                                  │
    │                  MPP编码+推流                                 │
```

**关键交互：**

| 步骤 | 操作 | 数据流向 |
|------|------|---------|
| 1 | Aggregator dequeue | `g_SafeQueueRead` → `inputFD` |
| 2 | `submit_task_async` | `inputFD` → `Task` → `tasks`（全局队列） |
| 3 | `task_cond.notify_one` | 唤醒一个 Worker |
| 4 | Worker 出队 | `tasks` → `Task t` |
| 5 | Worker 推理 | NV12→RGB(NPU) → rknn_run → 后处理 → BGR→NV12 |
| 6 | `promise.set_value(res)` | `res` → `future` ready |
| 7 | `wait_for(0)` 检测 | Aggregator 轮询 `nextWriteIndex` 的 future |
| 8 | `future.get()` | 获取 `ProcessResult` |
| 9 | `g_SafeQueueWrite.enqueue` | `outputFD` → WriteVideo |

## 5.9 线程退出顺序

```
STREAMON → ReadVideo 退出 → Aggregator 退出 → WriteVideo 退出
              │                    │                   │
          g_readFinish=true    g_processFinish=true    break
```

**原子变量传递链条：**

```
ReadVideo (采集结束)
    → g_readFinish = true
        → Aggregator 检测到 + 队列空 + tasks_inflight 空 → break
            → g_processFinish = true
                → WriteVideo 检测到 + 写队列空 → break
```

**SafeQueue 的 `stop()` 是兜底机制**（`main.cpp:649-650`），在线程 join 前调用，确保阻塞在 `dequeue` 上的线程能被唤醒退出。

## 5.10 旧架构 Aggregator 特性总结

| 特性 | 旧架构做法 |
|------|-----------|
| **任务提交** | `submit_task_async()` 返回 `future<ProcessResult>` |
| **Worker 分配** | 全局 `SharedQueue<Task>`，所有 Worker 竞争取任务 |
| **结果收集** | `future.wait_for(0)` 轮询 + `future.get()` |
| **保序方式** | `map<int, future>>` + `nextWriteIndex` |
| **队头阻塞** | 存在——慢帧阻塞后续所有帧 |
| **丢帧机制** | 无——所有帧都提交推理 |
| **NV12 拷贝** | ReadVideo 中 malloc+memcpy 一次，Worker 中 BGR→NV12 再 malloc 一次 |
| **线程同步** | `mutex` + `condition_variable`（全局任务队列锁竞争） |
| **C++ 特性** | `std::promise`, `std::future`, `std::move` |
