# 面试问答：零拷贝数据通路

## Q1: 代码中有一段 malloc + memcpy，这是否违反了"零拷贝"的设计原则？

### 回答

**当前代码中已经没有 malloc + memcpy 了。** 旧版中 ReadVideo 线程确实有一次 CPU 参与的 NV12 拷贝（DMABUF → 堆内存），但在重构后的方案 B（静态路由 + 零拷贝直写）中，这唯一一次 CPU 拷贝已被完全消除。

### 旧版架构回顾

旧版中，ReadVideo 线程从 V4L2 出队一帧 NV12 后，由于 DMABUF 缓冲区必须 QBUF 归还给 ISP 驱动循环使用，数据被 `malloc + memcpy` 到堆内存：

```
ISP(DMABUF) → [mapped_addr] → malloc(1.3MB) + memcpy → heap_copy → Worker
```

这次拷贝是架构约束下的解耦手段——Worker 不能阻塞 ReadVideo 的 QBUF 操作。

### 新版架构：如何消除这次拷贝

重构的核心思路是 **将 RGA 缩放+色彩转换从 Worker 移到 ReadVideo 线程**，且源从 CPU heap 改为 DMABUF fd：

```
ReadVideo 线程:
  V4L2 DQBUF → DMABUF fd (NV12 1280×720)
    ├── Path A: RGA importbuffer_fd(DMABUF) → imcvtcolor(NV12→BGR) → cv::Mat (画框用)
    └── Path C: RGA importbuffer_fd(DMABUF) → imresize(NV12→RGB 640x640)
                 → importbuffer_fd(npu_fd)   ← 直写 NPU 物理内存，CPU 零介入
```

关键突破：
1. **NPU 内存 fd 全局可见**：重构前 NPU 输入内存（`rknn_tensor_mem.fd`）在 Yolov5s 构造函数内部分配，ReadVideo 无法获取，被迫用 `malloc+memcpy` 中转
2. **重构后**：ReadVideo 在 RGA 写入前通过 `get_worker_input_fd()` 拿到 NPU 内存 fd，RGA 直接从 DMABUF fd 读取 NV12，缩放+转换后直写 NPU 物理内存
3. **一次 `imresize` 完成三件事**：缩放（1280×720→640×640）+ 色彩转换（NV12→RGB）+ DMA 直写 NPU

NV12 数据只在 DMABUF 中存活（DQBUF→QBUF 窗口），RGA 在这个窗口内完成硬件读取，不需要 CPU 介入。

### 真正的零拷贝路径

```
ISP → [DMABUF] → RGA 直接读 fd → NPU 物理内存   ✓ 全硬件链路
RGA 从 DMABUF fd 读 NV12 → 直写 NPU 内存       ✓ RGA 硬件 DMA，CPU 零介入
NPU → output_mems[i]->virt_addr                ✓ 推理结果直接 CPU 读取
```

### 面试中完整的表述

> 我的项目中实现了从 IMX415 摄像头到 NPU 推理的全链路零拷贝。核心思路是 **NPU 内存 fd 全局共享 + RGA 硬件直写**。ReadVideo 线程在 V4L2 DQBUF 拿到 DMABUF 后，立即通过 RGA `importbuffer_fd` 读取 NV12 并 `imresize` 为 RGB 640×640，再通过 `importbuffer_fd(npu_fd)` 直接写入 NPU 物理内存。整个过程 RGA 硬件 DMA 完成缩放+转换+搬运，CPU 不触碰任何像素数据。**这是真正的全链路零拷贝——ISP→RGA→NPU 全部通过 DMA 硬件搬运，CPU 只做调度触发。**

## Q2: 讲下你的零拷贝数据通路

### 推荐回答：四段式

按数据流分 4 段讲，每段一句话，让面试官跟着你的数据走一遍：

---

#### 第一段：Sensor → ISP → DMABUF

> IMX415 sensor 通过 MIPI CSI-2 4-lane 输出 RAW10 给 RKAIQ ISP，ISP 硬件完成 demosaic、3A、CCM、gamma 整条管线，直接输出 NV12 写入 V4L2 的 DMABUF 缓冲区。**整个过程 CPU 零参与，sensor 和 ISP 之间是硬件直连。**

#### 第二段：DMABUF → RGA → BGR（画框）

> ReadVideo 线程从 V4L2 出队拿到 DMABUF fd，RGA 通过 `importbuffer_fd` 直接读取 DMABUF 中的 NV12 数据，硬件转换为 BGR 用于 OpenCV 画框。**这里 RGA 是 DMA 设备，直接从物理内存读，不需要 CPU 把数据拷贝到用户空间。** 耗时约 3ms，纯 CPU 实现需要 ~30ms。

#### 第三段：DMABUF → RGA 直写 NPU（关键段 — True Zero-Copy）

> ReadVideo 线程通过 round-robin 计算目标 Worker（`index % 4`），检查 `is_busy` 确保 NPU 缓冲区空闲，然后通过 RGA 直接从 DMABUF fd 读取 NV12，`imresize` 为 RGB 640×640 后通过 `importbuffer_fd(npu_fd)` 直接写入 NPU 物理内存。**这次 RGA 操作同时完成了 1280×720→640×640 缩放、NV12→RGB 色彩转换、以及 DMA 直写 NPU 物理内存，CPU 只负责触发 RGA 硬件，不搬运任何像素。** 这就是项目真正的零拷贝核心——ISP→RGA→NPU 全链路硬件直传。

#### 第四段：NPU 推理 → 后处理 → 绘制 → MPP 编码

> RKNN 推理通过 `rknn_create_mem` 绑定输入输出物理内存，推理结果通过 `output_mems[i]->virt_addr` 由 CPU 读取 INT8 检测元数据，做反量化和 NMS。得到的检测框画到 BGR 图像上，再经 RGA `imcvtcolor` 转回 NV12 送 MPP 硬件编码 H.264，最后 RTMP 推流。

---

### 四条路径的零拷贝分析

| 路径 | 方式 | 是否零拷贝 |
|------|------|-----------|
| ISP → DMABUF | 硬件直接写入，CPU 不参与 | **零拷贝** |
| DMABUF → BGR（画框） | RGA `importbuffer_fd` 直接从 DMABUF fd 读取 | **硬件加速，零 CPU 拷贝** |
| DMABUF → NPU 输入 | RGA `importbuffer_fd(DMABUF)` → `imresize` → `importbuffer_fd(NPU)` 硬件直写 | **全硬件链路，零 CPU 拷贝** |
| NPU 推理 → 后处理| `rknn_create_mem` 零拷贝绑定   | **零拷贝** |

---

### 面试官可能的追问

#### Q: 你们怎么做到全链路零拷贝的？关键突破是什么？

> 关键突破是将 **NPU 内存 fd 全局可见化**。旧版中 `rknn_create_mem` 在 Yolov5s 构造函数内部调用，RGA 无法获取 NPU 内存的 fd，被迫用 `malloc+memcpy` 中转。重构后 ThreadPool 提供 `get_worker_input_fd()` 方法，ReadVideo 通过 round-robin 选定 Worker 后直接拿到该 Worker 的 NPU 内存 fd，RGA 直接写入，完全消除 CPU 拷贝。

#### Q: 没有 CPU memcpy 后，怎么解决 DMABUF 被 QBUF 回收的问题？

> RGA 是同步阻塞的（`imresize` 在数据搬完后才返回），所以在 DQBUF→RGA 搬运→QBUF 这个窗口内，RGA 已通过 DMA 把数据从 DMABUF 读到了 NPU 物理内存。QBUF 后 ISP 覆盖的是旧数据，NPU 内存中有独立的副本。这实际上是用 **RGA 硬件 DMA 替代了 CPU memcpy**——数据搬移的逻辑没变，但搬运者从 CPU 换成了 RGA DMA 引擎。

#### Q: 你这个方案相比 TensorRT、MediaPipe 有什么优劣？

> 优势是对 Rockchip 硬件利用率高，ISP、RGA、NPU、MPP 每层硬件都在做它们最擅长的事，没有 CPU 瓶颈。劣势是平台绑定强，换平台全部重写。成熟方案通用性好，但硬件细节封装在底层，无法做到这种颗粒度的优化。

---

## 一句话总结

> **像素路径零拷贝，元数据路径 CPU 读取。** — 这是嵌入式 AI 视觉的典型模式。图像数据在 ISP→DMABUF→RGA→NPU→MPP 之间硬件直传，只有 NV12 拷贝到堆内存那一步是 CPU 参与的。


# 零拷贝架构演进：从 CPU memcpy 到全链路硬件 DMA

## 架构演进对比

### 旧版（方案 A）：CPU 参与搬运

```
ReadVideo: DMABUF → malloc+memcpy → heap_copy → SafeQueue
Worker:    heap_copy → RGA(NV12→RGB) → NPU 内存 → rknn_run
```

- ReadVideo 中一次 CPU `memcpy`（DMABUF → 堆内存）
- Worker 中 RGA 从 CPU 内存通过 `importbuffer_virtualaddr` 读取

### 新版（方案 B）：全链路零拷贝

```
ReadVideo: DMABUF → RGA(NV12→RGB) → NPU 物理内存 → SafeQueue（仅 BGR Mat）
Worker:    rknn_run  ← NPU 直接从 input_mem 读取
```

- CPU 不触碰任何像素数据
- ISP→RGA→NPU 全部通过 DMA 硬件搬运
- 用 `is_busy` 原子锁保护 NPU 缓冲区，避免多硬件并发的内存踩踏
