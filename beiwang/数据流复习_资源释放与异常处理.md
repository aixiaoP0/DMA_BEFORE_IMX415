## 8.1 线程退出链

三个线程以**链式顺序**退出：

```
STREAMON → ReadVideo 退出 → Aggregator 退出 → WriteVideo 退出
              │                    │                    │
          g_readFinish=true    g_processFinish=true    break
```

### 退出信号传递

```cpp
// main.cpp:155-158（全局变量）
std::atomic<bool> g_readFinish(false);     // ReadVideo 停止标志
std::atomic<bool> g_processFinish(false);  // Aggregator 停止标志
```

使用 `std::atomic<bool>` 保证多线程间可见性（memory ordering），不需要加锁。

### ReadVideo 退出条件

**相关代码：** `main.cpp:287-291`

```cpp
if (ioctl(fd, VIDIOC_QBUF, &buf) < 0) {
    perror("queue buf");
    finished = true;   // 设置退出标志
    break;
}
```

ReadVideo 只会在以下情况退出：

1. `poll` 失败（设备异常）
2. `DQBUF` 失败（驱动异常）
3. `QBUF` 失败（驱动异常）
4. `poll` 超时（暂时性无帧，`continue` 不退出）

即 ReadVideo 的退出**完全由硬件异常驱动**，没有"处理完 N 帧后主动退出"的逻辑。

### Aggregator 退出条件

**相关代码：** `main.cpp:367-371`

```cpp
if (g_readFinish && g_SafeQueueRead.empty() && tasks_inflight.empty()) {
    break;
}
```

三重条件**缺一不可**：

| 条件 | 含义 | 为什么需要 |
|------|------|-----------|
| `g_readFinish` | ReadVideo 已停止 | 否则还有新帧进来 |
| `g_SafeQueueRead.empty()` | 读队列中的残留帧已全部取出 | 否则还有帧未提交推理 |
| `tasks_inflight.empty()` | 所有已提交任务都已完成 | 否则推理结果未收集，直接退出会丢失结果+内存泄漏 |

### WriteVideo 退出条件

**相关代码：** `main.cpp:399-402`

```cpp
if (g_processFinish && g_SafeQueueWrite.empty()) {
    break;
}
```

需要写队列清空才退出，否则会丢失未编码推流的帧。

## 8.2 main() 中的等待顺序

**相关代码：** `main.cpp:619-624`

```cpp
// 先等 ReadVideo 退出
video_reader.join();

// 再等 WriteVideo 退出
for (thread& t : video_writer) { t.join(); }

// 最后等 Aggregator 退出
video_process.join();
```

**为什么 ReadVideo 先 join？**

Aggregator 依赖 `g_readFinish` 信号来判断退出，所以 ReadVideo 必须先退出、设置标志，Aggregator 才有机会退出。但这里 `video_reader.join()` 是阻塞等待 ReadVideo 完全结束才开始下面的 join。

**这里有个隐含假设：** 如果 ReadVideo 一直不退出（比如一直有帧），Aggregator 和 WriteVideo 永远不会退出，main() 就会阻塞在 `video_reader.join()`。在 frame_num=2000 的设定下这是预期行为——ReadVideo 采完 2000 帧后会因为 V4L2 驱动停止或外部通知而退出。但在无限采集模式下需要有信号处理机制来中止。

> **设计不足：** 当前代码**没有注册信号处理器**（如 SIGINT/Ctrl+C）。如果运行时需要中止，三个线程无法优雅退出，资源释放路径不会执行。生产环境中应添加信号处理，设置 `g_readFinish = true` 并唤醒阻塞的 `dequeue`（通过 SafeQueue 的 `stop()`）。

## 8.3 SafeQueue stop() 机制

**相关代码：** `SafeQueue.h:46-51`

```cpp
void stop() {
    unique_lock<mutex> lock(m);
    stop_flag = true;
    cond_not_empty.notify_all();
    cond_not_full.notify_all();
}
```

当 `stop()` 被调用后，阻塞在 `dequeue` 上的线程会被唤醒，然后检查 `stop_flag`：

```cpp
// SafeQueue.h:35-39
bool dequeue(T& t) {
    unique_lock<mutex> lock(m);
    cond_not_empty.wait(lock, [this]{ return !q.empty(); });
    if (stop_flag && q.empty()) {
        return false;  // 停止信号，返回 false 让调用方退出
    }
    t = q.front();
    q.pop();
    cond_not_full.notify_all();
    return true;
}
```

`dequeue` 的退出逻辑：

```
cond_not_empty 被唤醒
    ↓
检查 predicate: `!q.empty()`
    ├── 队列还有数据 → 正常出队，返回 true
    └── 队列为空 AND stop_flag=true → predicate 为 false
         ↓
    wait() 返回，代码继续往下走
         ↓
    if (stop_flag && q.empty()) → return false
```

**调用时机：** `main.cpp:649-650`

```cpp
g_SafeQueueRead.stop();
g_SafeQueueWrite.stop();
```

在所有 `join()` 完成后调用，作为兜底安全措施。如果某个线程因异常没有正常退出（仍阻塞在 `dequeue` 上），`stop()` 能确保它被唤醒。

## 8.4 Worker 异常处理

**相关代码：** `thread_pool.cpp:188-194`

```cpp
catch(const std::exception& e) {
    res.error_msg = e.what();
    res.success = false;
    if(res.nv12_data) { free(res.nv12_data); res.nv12_data = nullptr; }
}
```

Worker 的异常保护策略：

- **try 块包裹整个处理逻辑**：从推理到绘制
- **异常时不崩溃线程**：Worker 线程继续存活，可以处理后续任务
- **确保内存释放**：仅释放 `res.nv12_data`（BGR→NV12 转换结果，输出编码用）
- **错误信息传递**：通过 `ProcessResult.error_msg` 传回 Aggregator

正常路径（BGR→NV12 转换 + 结果设置）：

```cpp
BGR_to_NV12_with_rga(t.img.data, pixel_buffer, t.width, t.height);
res.nv12_data = pixel_buffer;  // 所有权转移：Worker → ProcessResult → WriteVideo
res.processed_img = t.img;
res.data_size = nv12_size;
res.success = true;
```

> **重构变化：** 旧版中 `t.nv12_data` 需要 Worker 手动 `free`（输入 NV12 拷贝）。重构后输入 NV12 由 ReadVideo 的 RGA 硬件 DMA 直写 NPU 物理内存，不再有 malloc/free 的输入 NV12 内存。Worker 只需管理输出编码用的 NV12（`res.nv12_data`）。

## 8.5 线程池析构

**相关代码：** `thread_pool.cpp:28-45`

```cpp
ThreadPool::~ThreadPool() {
    this->run_flag = false;

    // 唤醒所有 Worker（每个 Worker 自己的条件变量）
    for(size_t i = 0; i < threads.size(); i++) {
        worker_ctx[i].cv.notify_all();
    }

    // 等待所有工作线程退出
    for (thread &t : threads) {
        if (t.joinable()) {
            t.join();
        }
    }
}
```

```cpp
// Worker 退出检查
worker_ctx[id].cv.wait(lock, [this, id]{
    return (!worker_ctx[id].tasks.empty() || !run_flag);
});
if (!run_flag && worker_ctx[id].tasks.empty()) {
    break;  // 线程池停止 + 无剩余任务 → 退出
}
```

> **重构变化：** 旧版使用单一 `task_cond` 通知所有 Worker，新版使用 `worker_ctx[i].cv` 每个 Worker 独立条件变量。析构时通过索引循环 `for(size_t i = 0; i < threads.size(); i++)` 逐一通知（`unique_ptr<WorkerContext[]>` 不支持 range-for）。

## 8.6 内存释放链

### NV12 数据所有权（重构后简化）

```
ReadVideo               Aggregator               Worker                WriteVideo
  │                        │                       │                      │
  │  Path C: RGA 直写 NPU  │                       │                      │
  │  (无需 malloc NV12)    │                       │                      │
  │                        ├── submit_to_worker     │                      │
  │                        │   (无 NV12 数据)       │                      │
  │                        │                       │                      │
  │                        │                       ├── malloc(nv12)       │
  │                        │                       │    (BGR→NV12 结果)   │
  │                        ├── collect result       │                      │
  │                        │   (move nv12_data)     │                      │
  │                        │                       │                      │
  │                        ├── enqueue ────────────┼──────────────────────┤
  │                        │                       │                      ├── free(nv12)
  │                        │                       │                      │    (编码推流后)
```

Key 规则：**谁分配谁释放**——重构后只有一段 NV12 内存：

| 内存块 | 分配者 | 使用者 | 释放者 |
|--------|--------|--------|--------|
| NV12 编码 | Worker (`malloc`) | WriteVideo (MPP 编码) | WriteVideo (`free`) |
| BGR 帧 | ReadVideo (`clone`) | Aggregator → Worker (绘制) | Worker (局部变量自动释放) |

> **重构收益：** 旧版有两段 NV12 内存（输入拷贝 + 输出编码），重构后输入 NV12 拷贝被完全消除——ReadVideo 通过 RGA 硬件 DMA 直接将 NV12→RGB 写入 NPU 物理内存，不再需要 malloc/free。系统堆分配从每帧 2 次减少到每帧 1 次（仅编码用 NV12）。

### DMABUF 清理

**相关代码：** `main.cpp:633-645`

```cpp
// 1. 逐个清理 DMABUF 缓冲区（unmap + close(fd)）
for (int i = 0; i < state.num_buffers; i++) {
    my_dmabuf_buffer_cleanup(&state.dmabuf_buffer[i]);
}
free(state.dmabuf_buffer);

// 2. 清理 V4L2 buffer_info
free(state.buffer_infos);

// 3. 关闭 DMA 堆
my_dmabuf_heap_cleanup(&state.dmabuf_heap);
```

`my_dmabuf_buffer_cleanup` 的内部顺序（`dmabuf.c:258-278`）：

```
1. my_dmabuf_buffer_unmap()  → munmap 解除映射
2. close(fd)                 → 关闭 DMABUF 文件描述符
3. memset(buffer, 0)         → 重置结构体
```

> **为什么先 unmap 再 close？** unmap 解除的是用户空间的虚拟地址映射，close 释放的是内核中的 DMABUF 引用计数。必须先切断 CPU 访问路径，再释放内核资源。

### 推流模块清理

**相关代码：** `streamer.c:132-150`

## 8.7 完整清理序列

```
main() 退出时执行顺序：

1. video_reader.join()     → 等 ReadVideo 退出（自然或因 QBUF 失败）
2. video_writer[0].join()  → 等 WriteVideo 退出
3. video_process.join()    → 等 Aggregator 退出
                           ↓
4. VIDIOC_STREAMOFF        → 停止 ISP 数据流
5. my_dmabuf_buffer_cleanup × N  → 逐个释放 DMABUF
6. my_dmabuf_heap_cleanup  → 关闭 dma_heap
7. close(fd)               → 关闭 V4L2 设备
8. g_SafeQueueRead.stop()  → 唤醒残留阻塞线程
9. g_SafeQueueWrite.stop()
                           ↓
10. writer.release()       → 释放 OpenCV VideoWriter
```

## 8.8 现有设计不足

| 问题 | 影响 | 改进方向 |
|------|------|---------|
| 没有 SIGINT 信号处理 | Ctrl+C 无法优雅退出，资源可能泄漏 | 注册 `signal(SIGINT, handler)` 设置 `g_readFinish=true` |
| Aggregator 中没有任务超时机制 | 某个帧推理无限阻塞会导致整个管线停止 | `wait_for` 超时 + 跳帧策略 |
| ReadVideo 无帧数上限控制 | 无限采集模式下无法主动停止 | 添加 frame_num 检查 |
| WriteVideo 空洞等待 | 队列为空时 sleep(2ms)，退出响应延迟 | 使用条件变量等待 |
