# 零拷贝数据通路重构：RK3588 NPU 推理管线架构演进

---

## 1. 背景与重构目标

### 1.1 现有架构的痛点分析

原始管线在 RK3588 上实现了"IMX415 → ISP → YOLOv5s NPU → RTMP"的基本流程，但数据通路中存在多处 CPU 介入的瓶颈：

**痛点一：malloc + memcpy 的像素级搬运**

ReadVideo 线程从 V4L2 DMABUF 出队一帧 NV12（1280×720）后，由于 DMABUF 缓冲区必须 `QBUF` 归还给 ISP 驱动循环使用，数据必须拷贝出来：

```
ISP(DMABUF) → [mapped_addr] → malloc(1.3MB) + memcpy → heap_copy → Worker
```

每帧 1.3MB，30fps 下每秒 40MB 的纯 CPU 搬运。ARM 的 CPU 做这件事有三重代价：

- **内存带宽耗尽**：40MB/s 的像素搬运挤占了本可作为他用（后处理、编码）的总线带宽
- **Cache 污染**：memcpy 流经 L1/L2 Cache，而像素数据是一次性消费品，冲刷掉 Cache 中的热点代码和常用变量
- **非确定性延迟**：malloc 在长时间运行后会产生碎片，分配延迟不可预测

**痛点二：生产者与消费者的内存耦合**

NPU 输入内存（`rknn_tensor_mem`）在 `Yolov5s` 构造函数内部通过 `rknn_create_mem()` 分配，作用域局限在 Worker 线程内部。ReadVideo 线程无法获取 NPU 内存的 `fd`，被迫用 malloc+memcpy 中转：

```
生产者(ReadVideo) 握有 V4L2 DMABUF fd，但摸不到 NPU 内存
消费者(Worker)     拥有 NPU 内存 fd，但数据是 CPU memcpy 进来的
```

**痛点三：多线程调度层的性能瓶颈**

- `std::future`/`std::promise` 机制：每次任务提交都涉及一次堆分配（promise 状态存储）和一次原子同步（future 就绪通知），Aggregator 通过 `wait_for(0)` 轮询产生不必要的上下文切换
- 单共享任务队列：4 个 Worker 竞争同一个 `mutex` 保护的队列，锁争用在高帧率下显著
- `cv::Mat::clone()` 深拷贝：Aggregator 中无意义的 2.7MB/帧深拷贝（1280×720×3 BGR），WriteVideo 只读 `nv12_data`，`frame` 被克隆后立即丢弃

### 1.2 架构重构目标

1. **True Zero-Copy（全链路零拷贝）**：CPU 不触碰任何像素数据，数据在 ISP→RGA→NPU 之间全部通过 DMA 硬件搬运
2. **解除生产-消费耦合**：NPU 内存 fd 全局可见，ReadVideo 通过 RGA 直接写入 NPU 物理内存
3. **无锁化调度**：每 Worker 私有队列 + 无 future 结果收集，消除锁竞争和堆分配开销
4. **工业级健壮性**：原子锁防内存踩踏 + 主动丢帧 + 严格保序

## 2. 整体架构演进

### 2.1 旧版数据流链路（CPU 参与的搬运）

```
ReadVideo 线程:
  V4L2 DQBUF → DMABUF fd (NV12 1280×720)
    ├── Path A: RGA importbuffer_fd(DMABUF fd) → imcvtcolor(NV12→BGR) → cv::Mat
    └── Path B: malloc(nv12_size) + memcpy(mapped_addr) → heap_copy

SafeQueue (maxSize=100)

Worker i 线程:
    ├── Path C: importbuffer_virtualaddr(heap_copy) → imresize(NV12→RGB 640x640)
    │            → importbuffer_fd(npu_fd)   ← RGA 从 CPU 内存读取，非零拷贝
    ├── rknn_run()
    ├── CPU: post_process + draw
    └── Path D: RGA BGR→NV12 (for encoder)

结果: std::future<ProcessResult> → Aggregator 轮询
```

**问题总结**：每帧经历 3 次不必要的像素搬运：ISP→(DMA)→CPU(memcpy)→RGA→NPU。

### 2.2 新版数据流链路（方案 B：静态路由 + 零拷贝直写）

```
ReadVideo 线程:
  V4L2 DQBUF → DMABUF fd (NV12 1280×720)
    ├── Path A: RGA importbuffer_fd(DMABUF) → imcvtcolor(NV12→BGR) → cv::Mat  (画框用)
    └── Path C: RGA importbuffer_fd(DMABUF) → imresize(NV12→RGB 640x640)
                 → importbuffer_fd(npu_fd)   ← 直写 NPU 物理内存，CPU 零介入

SafeQueue (maxSize=100)

  ↓ round-robin (index % 4)

Worker i 线程:
    ├── rknn_run()  ← NPU 直接从 input_mem 物理地址读取
    ├── CPU: post_process + draw
    └── Path D: RGA BGR→NV12 (for encoder)

结果: result_queue (无 future) → Aggregator map 重排
```

**关键变化**：
1. Path C 从 Worker 移入 ReadVideo，且源从 CPU heap 改为 DMABUF fd，实现纯硬件 ISP→RGA→NPU 链路
2. 去掉 malloc+memcpy，每帧节省 1.3MB 堆分配 + 拷贝
3. 去掉 std::future 机制，结果通过轻量级队列回传

### 2.3 线程交互与流水线模型

```
┌───────────────────────────────────────────────────────────────────────────┐
│  main()                                                                   │
│  V4L2 init → ThreadPool(4 workers) → 启动三线程                            │
└───────────────────────────────────────────────────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
  ReadVideo            Aggregator          WriteVideo
  (采集+Path A+Path C)  (路由+保序)          (编码+推流)
        │                   │                   │
        │  g_SafeQueueRead  │                   │
        ├──────────────────►│                   │
        │   (max 100帧)     │                   │
        │                   │  round-robin      │
        │                   ├──┬──┬──┬──┐       │
        │                   │W0│W1│W2│W3│       │
        │                   ├──┴──┴──┴──┤       │
        │                   │ result_q  │       │
        │                   │ map重排    │       │
        │                   │           │       │
        │                   │ g_SafeQueueWrite  │
        │                   ├──────────────────►│
        │                   │                   │
        │                   │                   │  MPP → RTMP
```

三线程职责：
| 线程 | 函数 | 职责 |
|------|------|------|
| **ReadVideo** | `Thread_ReadVideo()` | V4L2 poll 采集 → Path A(BGR 画框用) + Path C(RGA 直写 NPU) |
| **Aggregator** | `aggregatorThreadFunc()` | Round-robin 分发 → 收集结果 → map 按序写入输出队列 |
| **WriteVideo** | `Thread_WriteVideo()` | MPP 编码 H.264 → RTMP 推流 → 每 30 帧打印 FPS |

4 个 Worker 线程（`ThreadPool::worker`）：
| Worker | CPU 核心 | NPU 核心 | 职责 |
|--------|----------|----------|------|
| 0 | CPU4 | Core 0 | 推理 + 画框 + BGR→NV12 |
| 1 | CPU5 | Core 1 | 同上 |
| 2 | CPU6 | Core 2 | 同上 |
| 3 | CPU7 | Core 0（复用） | 同上 |

## 3. 核心机制设计与难点攻克

### 3.1 跨硬件 IP 的底层内存共享机制

实现全链路零拷贝的核心媒介是 **DMABUF（DMA Buffer）文件描述符**。

**DMABUF fd 的跨硬件流转**：

```
                    ┌─────────────────────────────────────┐
                    │            Linux Kernel              │
                    │   CMA (Contiguous Memory Allocator)  │
                    │          物理内存页框                 │
                    └─────────────────────────────────────┘
                                ▲
                   ┌────────────┼────────────┐
                   │            │            │
              ┌────┴────┐ ┌────┴────┐ ┌────┴────┐
              │  V4L2   │ │  RGA    │ │  NPU    │
              │ Driver  │ │ Driver  │ │ Driver  │
              └─────────┘ └─────────┘ └─────────┘
                   │            │            │
              dmabuf_fd    importbuffer_fd  rknn_create_mem
                   │            │            │
                   ▼            ▼            ▼
               ┌─────────────────────────────────┐
               │      物理内存 (同一块物理页)       │
               │  NV12 1280×720    RGB 640×640    │
               └─────────────────────────────────┘
```

关键 API 调用链：

```cpp
// 1. V4L2 通过 ISP DMA 将 NV12 写入 DMABUF（ReadVideo 中）
//    获取 dmabuf_fd = buffer_info.dmabuf_fd

// 2. RGA 从同一 fd 读取 NV12，写入 NPU 物理内存（Path C）
rga_buffer_handle_t src = importbuffer_fd(buffer_info.dmabuf_fd, nv12_size);
rga_buffer_handle_t dst = importbuffer_fd(npu_fd, 640 * 640 * 3);
// RGA 硬件 DMA 直接在两个物理地址之间搬运+缩放+色彩转换

// 3. NPU 从 input_mem 读取（Worker 中）
rknn_run(ctx, NULL);  // 无需额外 memcpy
```

`rknn_tensor_mem` 的三接口特性：

```
rknn_tensor_mem {
    .virt_addr     → CPU 侧 mmap 地址（此项目不用 CPU 写输入）
    .fd            → 文件描述符（RGA 通过 importbuffer_fd 硬件直写）
    .logical_addr  → NPU 内部地址（rknn_run 时 NPU DMA 读取）
    .size          → size_with_stride（硬件对齐后的大小）
}
```

同一块物理内存，三方通过不同接口访问：RGA 用 `fd` 写，NPU 用 `logical_addr` 读，CPU 用 `virt_addr` 读输出。

### 3.2 任务分发与静态路由轮询策略

**Round-Robin 派发**：

```cpp
// Aggregator 中
int target_worker = inputFD.index % g_num_workers;
gthreadpool.submit_to_worker(target_worker, inputFD.index, inputFD.frame, ...);
```

帧序 → Worker 映射：
```
Frame 0 → Worker 0    Frame 1 → Worker 1    Frame 2 → Worker 2    Frame 3 → Worker 3
Frame 4 → Worker 0    Frame 5 → Worker 1    Frame 6 → Worker 2    Frame 7 → Worker 3
...
```

**每 Worker 独立队列**消除锁竞争：

```cpp
// ThreadPool 内部
struct WorkerContext {
    std::queue<Task> tasks;
    mutex mtx;
    condition_variable cv;
    std::atomic<bool> is_busy{false};
};

std::unique_ptr<WorkerContext[]> worker_ctx;  // 每 Worker 一个
```

提交时直接入队目标 Worker 的私有队列，只锁该队列的 mutex，4 个 Worker 互不干扰：

```cpp
void ThreadPool::submit_to_worker(int worker_id, ...) {
    // ... 构造 Task ...
    {
        lock_guard<mutex> lock(worker_ctx[worker_id].mtx);
        worker_ctx[worker_id].tasks.push(std::move(t));
    }
    worker_ctx[worker_id].cv.notify_one();
}
```

### 3.3 致命缺陷防范：多硬件并发下的"内存踩踏"

**问题复现**：

```
时间轴:
T+0ms:   ReadVideo RGA 直写 Worker 0 的 NPU 物理内存 (Frame 0)
T+5ms:   ReadVideo 入队 Frame 0, 继续采集
T+33ms:  ReadVideo 捕获 Frame 4, target_worker = 4 % 4 = 0
         → 试图再次 RGA 直写 Worker 0 的 NPU 物理内存
T+20ms:  Worker 0 刚执行 rknn_run() 读取同一块内存...
         → ★ 内存踩踏：RGA 覆写新数据，NPU 读到半帧残影 ★
```

**破局方案：`is_busy` 原子硬件锁 + 主动丢帧**

```cpp
// ReadVideo 中 (thread_pool.h/cpp)
if (gthreadpool.is_worker_busy(target_worker)) {
    // Worker 还在用这块物理内存做推理 → 丢弃本帧
    frame_temp.skip_inference = true;
} else {
    gthreadpool.set_worker_busy(target_worker, true);
    // RGA 直写 NPU 物理内存
    // ...
    g_SafeQueueRead.enqueue(frame_temp);
}
```

```cpp
// Worker 完成后释放锁
worker_ctx[id].is_busy.store(false);
```

`std::atomic<bool>` 保证：
- **无锁安全性**：`store/load` 是 lock-free 的，不涉及系统调用
- **Memory Order 保证**：`load` 读到 `true` 意味着 Worker 确实持有该缓冲区
- **轻量级**：仅 1 字节状态，无堆分配

丢帧对推理准确率的影响：在 30fps、4 Worker 场景下，单 Worker 每 133ms 才被分配一帧，而单帧推理仅约 20ms，`is_busy` 几乎不会被触发。仅在极端突发负载时起保护作用。

### 3.4 异步并发下的严格保序机制

4 个 Worker 推理耗时存在微小波动（NPU 核心负载、Cache 命中率），Worker 2 可能比 Worker 1 先完成。但 RTMP 推流必须按时间戳顺序喂入。

**Aggregator 的 nextWriteIndex + map 重排**：

```cpp
int nextWriteIndex = 0;
std::map<int, ProcessResult> result_buffer;  // 按 index 排序

// 收集所有完成的结果
WorkerResult wr;
while (gthreadpool.try_get_result(wr)) {
    result_buffer[wr.index] = std::move(wr.result);
}

// 按序写出
auto it = result_buffer.find(nextWriteIndex);
while (it != result_buffer.end()) {
    g_SafeQueueWrite.enqueue(outputFD);
    result_buffer.erase(it);
    nextWriteIndex++;
    it = result_buffer.find(nextWriteIndex);
}
```

**丢帧保序的衔接**：

当 `is_busy` 触发丢帧时，ReadVideo 标记 `skip_inference=true` 仍将 FrameData 入队。Aggregator 检测到该标记，创建一个空白 `ProcessResult` 放入 `result_buffer`，`nextWriteIndex` 正常推进，不会卡死：

```cpp
if (inputFD.skip_inference) {
    ProcessResult blank;
    blank.success = false;
    result_buffer[inputFD.index] = std::move(blank);
}
```

WriteVideo 中 `nv12_data == nullptr` 的帧被自然跳过，不影响 FPS 统计。

## 4. 模块级重构实现细节

### 4.1 数据结构的精简与解耦

**Task 结构体**（从 6 字段减至 4 字段）：

```cpp
// 旧版
typedef struct Task {
    int index;
    cv::Mat img;
    uint8_t* nv12_data = nullptr;   // 移除
    int nv12_size = 0;               // 移除
    int width = 0;
    int height = 0;
    std::promise<ProcessResult> promise;  // 移除
} Task;

// 新版
typedef struct Task {
    int index = 0;
    cv::Mat img;
    int width = 0;
    int height = 0;
} Task;
```

**WorkerResult 结构体**（新增，替代 future）：

```cpp
typedef struct WorkerResult {
    int index;            // 帧序标识
    ProcessResult result; // 推理结果（含 nv12_data）
} WorkerResult;
```

**FrameData 结构体**（新增 `skip_inference` 字段）：

```cpp
struct FrameData {
    cv::Mat frame;
    int index;
    bool skip_inference = false;  // 丢帧标记
    // ...
};
```

### 4.2 Thread_ReadVideo 核心改造

```
Path A (保留): NV12→BGR 1280×720, cv::Mat 画框用
    importbuffer_fd(dmabuf_fd) → imcvtcolor(YCbCr_420_SP→BGR_888)

Path C (新增): NV12→RGB 640×640 → NPU 物理内存
    importbuffer_fd(dmabuf_fd) → imresize(YCbCr_420_SP→RGB_888)
    → importbuffer_fd(npu_fd)
```

Path C 的核心代码：

```cpp
int target_worker = frame_temp.index % 4;
int npu_fd = gthreadpool.get_worker_input_fd(target_worker);

rga_buffer_handle_t src_c = importbuffer_fd(buffer_info.dmabuf_fd, nv12_size);
rga_buffer_handle_t dst_c = importbuffer_fd(npu_fd, 640 * 640 * 3);

rga_buffer_t src = wrapbuffer_handle(src_c, 1280, 720, RK_FORMAT_YCbCr_420_SP);
rga_buffer_t dst = wrapbuffer_handle(dst_c, 640, 640, RK_FORMAT_RGB_888);

imresize(src, dst);  // 一次 RGA 调用完成缩放 + 色彩空间转换
```

**为什么 `imresize` 能做到 NV12→RGB 转换？**

RGA 2D 加速器的 `imresize()` 在源和目标格式不同时会自动插入色彩空间转换。输入是 NV12（`YCbCr_420_SP`），输出是 RGB888（`RGB_888`），RGA 硬件在缩放的同时执行 `YUV→RGB` 矩阵运算，零 CPU 开销。

### 4.3 纯粹的 NPU 触发器：ThreadPool::worker

重构后 Worker 从"图像预处理 + 推理 + 后处理"简化为"推理 + 后处理"：

```cpp
void ThreadPool::worker(int id) {
    // 绑核到 A76 大核
    CPU_SET(4 + (id % 4), &cpuset);
    pthread_setaffinity_np(..., &cpuset);

    while (run_flag) {
        // 1. 从私有队列取任务
        Task t;
        {
            unique_lock<mutex> lock(worker_ctx[id].mtx);
            worker_ctx[id].cv.wait(lock, ...);
            t = std::move(worker_ctx[id].tasks.front());
            worker_ctx[id].tasks.pop();
        }

        // 2. 推理（NPU 输入已被 ReadVideo 写满）
        yolo->inference_zero_copy(t.width, t.height, &res.detection_results);

        // 3. 画框
        yolo->draw_result(t.img, &res.detection_results);

        // 4. BGR→NV12（编码器输入）
        BGR_to_NV12_with_rga(t.img.data, pixel_buffer, t.width, t.height);

        // 5. 推结果
        result_queue.push({t.index, std::move(res)});
        worker_ctx[id].is_busy.store(false);  // 解锁 NPU 缓冲区
    }
}
```

**绑核策略**：
- RK3588 的 4 个 A76 大核（CPU4-7）分配给 4 个 Worker
- 4 个 A55 小核（CPU0-3）留给系统调度、ReadVideo、Aggregator、WriteVideo
- NPU Core 0 被 Worker 0 和 Worker 3 共享，RKNN 驱动内部串行化 `rknn_run`

### 4.4 汇聚层 Aggregator 无锁化结果收集

**剔除 std::future 的动机**：

| 对比维度 | std::future | result_queue |
|---------|-------------|--------------|
| 堆分配 | 每次 submit 一次 promise 状态分配 | 零额外分配 |
| 同步开销 | std::promise::set_value 含原子 CAS | mutex lock（已有队列锁） |
| 轮询开销 | wait_for(0) 系统调用 | 检查队列 front |
| 批次收集 | 必须逐帧检查 | 一次 drain 全部 |

**result_queue 实现**：

```cpp
// 线程池公共结果队列
std::queue<WorkerResult> result_queue;
mutex result_mtx;

// Worker 推
{
    lock_guard<mutex> lock(result_mtx);
    result_queue.push({t.index, std::move(res)});
}

// Aggregator 批量取
WorkerResult wr;
while (gthreadpool.try_get_result(wr)) {
    result_buffer[wr.index] = std::move(wr.result);
}
```

## 5. C++ 工程化与健壮性保障

### 5.1 并发对象的生命周期管理

**问题**：`std::mutex` 和 `std::condition_variable` 是**不可复制（non-copyable）且不可移动（non-movable）** 的。将它们放入 `std::vector` 并调用 `resize()` 会触发编译错误：

```cpp
// ❌ 编译错误：mutex 不可复制
std::vector<WorkerContext> worker_ctx;
worker_ctx.resize(num_threads);
```

**解法**：使用 `std::unique_ptr<WorkerContext[]>` 在堆上分配数组：

```cpp
// ✅ 编译通过
std::unique_ptr<WorkerContext[]> worker_ctx;
worker_ctx.reset(new WorkerContext[num_threads]);
```

`new T[N]` 在堆上连续分配 N 个默认构造的对象，不涉及移动或拷贝。`unique_ptr<T[]>::operator[]` 提供与原生数组相同的下标访问，所有 `worker_ctx[id]` 调用处不变。

### 5.2 异常安全与内存防泄漏

Worker 中 `BGR_to_NV12_with_rga` 需要分配 `pixel_buffer`：

```cpp
uint8_t* pixel_buffer = (uint8_t*)malloc(nv12_size);
if (pixel_buffer) {
    BGR_to_NV12_with_rga(t.img.data, pixel_buffer, t.width, t.height);
    res.nv12_data = pixel_buffer;  // 所有权移交
}
```

异常时的清理策略：

```cpp
catch (const std::exception& e) {
    res.error_msg = e.what();
    res.success = false;
    if (res.nv12_data) {
        free(res.nv12_data);  // 确保不泄漏
        res.nv12_data = nullptr;
    }
}
```

所有权链：

```
Worker::malloc(pixel_buffer)
  → res.nv12_data = pixel_buffer    // Worker → ProcessResult
    → outputFD.nv12_data = result.nv12_data  // ProcessResult → FrameData
      → WriteVideo::free(output_FD.nv12_data)  // FrameData → 释放
```

每帧的 NV12 数据从 Worker 分配开始，到 WriteVideo 编码后释放，所有权通过指针移交，不额外拷贝。

### 5.3 硬件处理的内存对齐陷阱

RGA 硬件要求图像宽度/高度按 **16 字节对齐**：

```cpp
#define ALIGN(x, a) (((x) + (a) - 1) & ~((a) - 1))
g_hor_stride = ALIGN(width, 16);   // 1280 → 1280 (已对齐)
g_ver_stride = ALIGN(height, 16);  // 720  → 720  (已对齐)
```

当前 1280×720 恰好是 16 的倍数，未对齐时 RGA `wrapbuffer_handle` 会用 stride 计算内存访问范围。如果实际 malloc 的大小按 `width × height × 1.5` 计算而 RGA 按 `stride × stride × 1.5` 写入，就会发生**堆内存越界**。

**通用的安全写法**：

```cpp
// 分配时用 stride 而非 width/height
int alloc_size = g_hor_stride * g_ver_stride * 3 / 2;
uint8_t* buf = (uint8_t*)malloc(alloc_size);

// RGA 导入用实际大小（stride 已含在 wrapbuffer 参数中）
rga_buffer_t yuv = wrapbuffer_handle(handle, g_hor_stride, g_ver_stride, ...);
```

### 5.4 线程优雅退出机制

```cpp
ThreadPool::~ThreadPool() {
    this->run_flag = false;

    // 唤醒所有 Worker
    for (size_t i = 0; i < threads.size(); i++) {
        worker_ctx[i].cv.notify_all();
    }

    // 等待线程退出
    for (thread& t : threads) {
        if (t.joinable()) t.join();
    }
}
```

Worker 退出路径：

```cpp
while (run_flag) {
    // ...
    worker_ctx[id].cv.wait(lock, [this, id] {
        return !worker_ctx[id].tasks.empty() || !run_flag;
    });
    if (!run_flag && worker_ctx[id].tasks.empty()) break;
    // ...
}
```

- `run_flag = false` + `notify_all()` 确保所有 Worker 从 `cv.wait` 醒来
- predicate 条件 `|| !run_flag` 防止虚假唤醒后漏判退出信号
- 处理完剩余任务后再退出，不丢帧

## 6. 性能预期与总结

### 预期收益

| 指标 | 改造前 | 改造后 | 改善幅度 |
|------|--------|--------|----------|
| CPU 像素搬运 | 40 MB/s（memcpy） | 0 | 100% 消除 |
| 每帧堆分配 | malloc+free × 2 | malloc+free × 1（编码器） | 50% 减少 |
| 锁竞争 | 单队列 4 Worker 争抢 | 4 独立队列无竞争 | ~0% 锁等待 |
| Future 开销 | 每帧 1 次 promise 分配 | 0 | 100% 消除 |
| Mat 深拷贝 | 2.7MB/帧 clone | 引用计数拷贝 | ~0 耗时 |

### 关键设计决策总结

| 决策 | 选型 | 原因 |
|------|------|------|
| 数据通路 | ISP→RGA→NPU 纯硬件 | 消除 CPU memcpy |
| 调度策略 | Round-Robin 静态路由 | 确定性调度，可预测性好 |
| 队列模型 | 每 Worker 独立队列 | 消除锁竞争 |
| 结果收集 | result_queue + map 重排 | 无 future 轻量回传 + 严格保序 |
| 内存保护 | atomic is_busy + 丢帧 | 防止 GPU 级流水线数据踩踏 |
| Worker 绑核 | CPU4-7 (A76) | 大核专供推理，避免调度到小核 |
| Mat 传递 | 引用计数浅拷贝 | 避免无意义深拷贝 |
| WorkerContext | unique_ptr<T[]> | 规避 mutex 不可拷贝限制 |
