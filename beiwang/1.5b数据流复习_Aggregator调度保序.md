# 第五阶段：Aggregator 调度保序

**相关代码：** `main.cpp:320-391`（`aggregatorThreadFunc` 完整函数）
**相关线程池：** `thread_pool.h/cpp`（`submit_to_worker`、`try_get_result`、`worker`）

```
Aggregator 本质：乱序执行 + 顺序提交引擎（out-of-order execution with in-order commit）
与 CPU Tomasulo 算法有相同设计哲学。
```

---

## 5.1 要解决什么问题

线程池 4 个 Worker 并行推理，各 Worker 处理速度不同：

```
ReadVideo 采集顺序:  frame0  frame1  frame2  frame3  frame4  frame5 ...
                        │       │       │       │       │       │
                        ▼       ▼       ▼       ▼       ▼       ▼
Round-Robin 分发:    W0      W1      W2      W3      W0      W1      ...
(按 index % 4)
                    完成快    完成慢    完成快    完成中    完成快    ...
                        │       │       │       │       │       │
                        ▼       ▼       ▼       ▼       ▼       ▼
自然完成顺序:         0, 2, 3, 1, 4, ...   ← 乱序！
```

如果没有保序，画面会出现 **帧序跳跃**。

> **与旧架构对比：** 旧架构通过 `future/promise` 实现异步结果回传，Aggregator 在每个 Worker 提交任务后拿到 `future<ProcessResult>`，通过 `wait_for(0)` 轮询完成状态。新架构移除 future/promise，改用 `submit_to_worker()`（无返回值）+ `result_queue` 批量收集 + `result_buffer` map 重排。

```cpp
    mutex mtx;                          // 队列互斥锁
    condition_variable cv;              // 通知 Worker 有新任务

vector<shared_ptr<Yolov5s>> yolo_group;        // 每 Worker 独立 YOLO 实例
```

## 5.3 Submit_to_Worker：怎么通知 Worker 干活

**位置：** `thread_pool.cpp:82-99`

```cpp
void ThreadPool::submit_to_worker(int worker_id, int index, const cv::Mat& img,
                                   int width, int height)
{
    Task t;
    t.index  = index;
    t.img    = img;       // cv::Mat 浅拷贝（引用计数），BGR 数据不复制
    t.width  = width;
    t.height = height;

    tasks_in_flight++;    // 在途任务 +1

    // 1. 入队到指定 Worker 的私有队列（加锁保护）
    {
        lock_guard<mutex> lock(worker_ctx[worker_id].mtx);
        worker_ctx[worker_id].tasks.push(std::move(t));
    }  // 锁在此释放

    // 2. 通知 Worker 有新任务
    worker_ctx[worker_id].cv.notify_one();
}
```

**关键设计点：**

| 要点 | 说明 |
|------|------|
| **无 future** | 直接入队后返回，不等待结果。Aggregator 不阻塞 |
| **无锁竞争** | 每个 Worker 独立队列，提交 W0 和提交 W1 不竞争同一把锁 |
| **cv::Mat 浅拷贝** | OpenCV 引用计数，BGR 像素数据不复制，仅复制头部 |
| **先解锁再 notify** | 避免唤醒的 Worker 立即因锁竞争被挂起（lock_guard 析构后锁已释放） |
| **tasks_in_flight 先加** | 在入队成功前就计数增加，保证限流判断准确 |

### Worker 怎么被唤醒的

Worker 线程在 `cv.wait()` 上睡眠：

```cpp
// thread_pool.cpp:157-158
unique_lock<mutex> lock(worker_ctx[id].mtx);
worker_ctx[id].cv.wait(lock, [this, id] {
    return !worker_ctx[id].tasks.empty() || !run_flag;
});
```

- **正常情况下**：`submit_to_worker` push 任务后 `cv.notify_one()` → Worker 从 wait 返回 → 检查 predicate（`tasks.empty()` 为 false）→ 拿到锁 → pop 任务
- **退出时**：`run_flag = false` + `cv.notify_all()` → predicate 为 true → 循环退出

**条件变量 wait + predicate 的双重保护：**
1. `wait` 返回说明收到 notify（或虚假唤醒）
2. predicate 再次检查：队列确实不为空，或者线程池要退出了
3. 如果是虚假唤醒但队列仍空，predicate 返回 false，继续 wait

## 5.4 Worker 线程完整逻辑

**位置：** `thread_pool.cpp:135-208`

```cpp
void ThreadPool::worker(int id)
{
    // 1. 绑核：绑定到 CPU4-7（A76 大核）
    cpu_set_t cpuset;
    CPU_ZERO(&cpuset);
    CPU_SET(4 + (id % 4), &cpuset);
    pthread_setaffinity_np(pthread_self(), sizeof(cpu_set_t), &cpuset);

    shared_ptr<Yolov5s> yolo = this->yolo_group[id];

    while(run_flag)
    {
        // 2. 从自己的私有队列取任务（阻塞等待）
        Task t;
        {
            unique_lock<mutex> lock(worker_ctx[id].mtx);
            worker_ctx[id].cv.wait(lock, [this, id] {
                return !worker_ctx[id].tasks.empty() || !run_flag;
            });
            if(!run_flag && worker_ctx[id].tasks.empty()) break;
            if(worker_ctx[id].tasks.empty()) continue;   // 虚假唤醒保护

            t = std::move(worker_ctx[id].tasks.front());
            worker_ctx[id].tasks.pop();
        }  // 锁释放，其他线程可继续往此队列提交

        // 3. NPU 推理（输入已被 ReadVideo 通过 RGA 直写）
        ProcessResult res;
        try {
            yolo->inference_zero_copy(t.width, t.height, &res.detection_results);

            // 4. 绘制检测框
            yolo->draw_result(t.img, &res.detection_results);

            // 5. BGR → NV12（路径 D：RGA 硬件转换，供 MPP 编码器用）
            int nv12_size = t.width * t.height * 3 / 2;
            uint8_t* pixel_buffer = (uint8_t*)malloc(nv12_size);
            if(pixel_buffer) {
                BGR_to_NV12_with_rga(t.img.data, pixel_buffer, t.width, t.height);
                res.processed_img = t.img;
                res.nv12_data = pixel_buffer;
                res.data_size = nv12_size;
                res.success = true;
            }
        }
        catch(const std::exception& e) { /* 异常处理 */ }

        // 6. 结果入公共收集队列
        {
            lock_guard<mutex> lock(result_mtx);
            result_queue.push({t.index, std::move(res)});
        }  // 结果对 Aggregator 可见

        tasks_in_flight--;                    // 在途任务 -1

        // 7. 释放 NPU 缓冲区占用标志
        worker_ctx[id].is_busy.store(false);  // ReadVideo 可写新帧
    }
}
```

**逐个步骤分析：**

| 步骤 | 操作 | 含义 |
|------|------|------|
| 1 | 绑核 | `CPU_SET(4 + id % 4)`，Worker i 绑定到 CPU i+4（4 个 A76 大核） |
| 2 | cv.wait | 条件变量阻塞等待任务，不占 CPU。只在有任务或退出时被唤醒 |
| 3 | `inference_zero_copy` | NPU 已通过 Path C 被 RGA 写入输入数据，直接 `rknn_run` |
| 4 | `draw_result` | `cv::rectangle` + `cv::putText` 画框上 BGR 原图 |
| 5 | `malloc + RGA BGR→NV12` | **新架构唯一一次 NV12 堆分配**，编码器输入 |
| 6 | `result_queue.push` | 结果入公共队列，Aggregator 通过 `try_get_result` 收集 |
| 7 | `is_busy = false` | 通知 ReadVideo：NPU 缓冲区已空闲，可写入下一帧 |

### Worker 退出路径

```
循环条件: while(run_flag)
  → cv.wait 被 notify_all 唤醒
  → predicate: !tasks.empty() || !run_flag → true（run_flag=false）
  → tasks.empty() → true（无残留任务）
  → continue 跳过 → 回到 while 判断
  → run_flag = false → 退出循环
```

如果有残留任务，Worker 会在继续处理完再退出。

## 5.5 Try_get_result：怎么收集结果

**位置：** `thread_pool.cpp:116-123`

```cpp
bool ThreadPool::try_get_result(WorkerResult& wr)
{
    lock_guard<mutex> lock(result_mtx);
    if(result_queue.empty()) return false;
    wr = std::move(result_queue.front());
    result_queue.pop();
    return true;
}
```

**设计要点：**

| 要点 | 说明 |
|------|------|
| **非阻塞** | 队列空时返回 false，不等待。Aggregator 不会卡在这里 |
| **批量 drain** | Aggregator 在 `while(try_get_result(wr))` 中不断调用，直到队列空 |
| **std::move** | 避免 `ProcessResult` 中的 `cv::Mat` 和 `nv12_data` 的深拷贝 |
| **一把公共锁** | 所有 Worker 共享一把 `result_mtx`。4 Worker 竞争不频繁（写完才 push） |

**与旧架构 future 的对比：**

| 操作 | 旧架构（future） | 新架构（result_queue） |
|------|----------------|----------------------|
| 提交 | `promise` 堆分配 → `future` 返回 | per-worker 队列 push（无分配） |
| 结果回传 | `promise.set_value(res)` 原子状态变更 | `result_queue.push`（mutex 锁） |
| 结果获取 | `future.wait_for(0)` 轮询 + `future.get()` | `try_get_result` 批量 drain |
| 每帧开销 | 1 次堆分配 + 1 次原子同步 + 2 次轮询 | 1 次互斥锁操作（push）+ 1 次互斥锁操作（pop） |

## 5.6 Aggregator 三步循环

**位置：** `main.cpp:328-388`

```cpp
while(true) {
    bool is_idle = true;

    // ── 步骤 A：批量提交 ──
    while(!g_SafeQueueRead.empty() && gthreadpool.in_flight_count() < 20) {
        if(g_SafeQueueRead.dequeue(inputFD)) {
            if(inputFD.skip_inference) {
                // 丢帧：空白结果进 result_buffer，维持保序
                ProcessResult blank;
                blank.success = false;
                result_buffer[inputFD.index] = std::move(blank);
            } else {
                gthreadpool.submit_to_worker(
                    inputFD.index % 4,    // round-robin 选定 Worker
                    inputFD.index, inputFD.frame,
                    inputFD.width, inputFD.height);
            }
        }
        is_idle = false;
    }

    // ── 步骤 B：批量收集 ──
    WorkerResult wr;
    while(gthreadpool.try_get_result(wr)) {
        result_buffer[wr.index] = std::move(wr.result);
        is_idle = false;
    }

    // ── 步骤 C：按序写出 ──
    auto it = result_buffer.find(nextWriteIndex);
    while(it != result_buffer.end()) {
        FrameData outputFD;
        outputFD.index = nextWriteIndex;
        outputFD.frame = it->second.processed_img;
        outputFD.nv12_data = it->second.nv12_data;
        outputFD.data_size = it->second.data_size;

        g_SafeQueueWrite.enqueue(outputFD);  // 可能阻塞（下游慢时）

        result_buffer.erase(it);
        nextWriteIndex++;
        it = result_buffer.find(nextWriteIndex);
        is_idle = false;
    }

    // ── 退出检测 ──
    if(g_readFinish && g_SafeQueueRead.empty() &&
       gthreadpool.in_flight_count() == 0 && result_buffer.empty())
        break;

    if(is_idle) sleep_for(1ms);  // 无任何工作可做时休眠 1ms
}
```

### 步骤 A 详解

**提交条件：** `g_SafeQueueRead` 非空 **且** `in_flight_count < 20`

**Round-Robin 分发：** `inputFD.index % 4`

| Worker | 处理的帧序号序列 | 间隔 |
|--------|----------------|------|
| W0 | 0, 4, 8, 12, 16, ... | 4 × 33ms = 133ms |
| W1 | 1, 5, 9, 13, 17, ... | 133ms |
| W2 | 2, 6, 10, 14, 18, ... | 133ms |
| W3 | 3, 7, 11, 15, 19, ... | 133ms |

**`skip_inference` 处理：** ReadVideo 中已通过 `is_busy` 判断决定是否丢帧，标记为丢帧的帧在此处直接创建空白 `ProcessResult` 进 `result_buffer`，不经过 Worker。

**限流 `in_flight_count < 20`：** 控制在途任务数，防止 result_buffer 暴涨。见 5.9 节详细分析。

### 步骤 B 详解

**批量 drain：** `while(try_get_result(wr))` 一次循环可取出多个 Worker 的多个已完成结果。

**写入 result_buffer：** `result_buffer[wr.index] = std::move(wr.result)` — 结果尚未按序写出，先缓存。

**为什么不是直接写到 g_SafeQueueWrite？** 因为结果到达顺序 ≠ 帧序。A 帧可能先完成，B 帧后完成，但 B 帧序号比 A 小。必须等前序帧凑齐。

### 步骤 C 详解

**条件式写出：** 只写 `nextWriteIndex` 对应的帧，不存在的则等待下一轮。

**连续写出优化：** 用 `while` 而非 `if`，如果帧 5 和帧 6 都已在 buffer 中，一口气写两个。

**阻塞点：** `g_SafeQueueWrite.enqueue(outputFD)` — 当 WriteVideo 推流慢、写队列满 100 时，此处会阻塞在条件变量上。这是管线反压传播的关键路径。

### 空闲检测

```cpp
bool is_idle = true;   // 每轮循环初始化为 true
// 步骤 A/B/C 任一执行了工作 → is_idle = false
if(is_idle) sleep_for(1ms);
```

- 三步都没有工作 → 休眠 1ms 避免 busy-wait 空转 CPU
- 1ms 的选择：短于 1 帧间隔（33ms）但不至于频繁唤醒

## 5.8 is_busy 丢帧机制

### ReadVideo 侧的丢帧决策（main.cpp:249-260）

```cpp
int target_worker = frame_temp.index % g_num_workers;

// 检查目标 Worker 的 NPU 缓冲区是否被占用
if (gthreadpool.is_worker_busy(target_worker)) {
    frame_temp.skip_inference = true;   // 丢帧：RGA 不写 NPU，不提交推理
    printf("---触发丢帧---帧[%d]已丢，目标Worker[%d]\n",
           frame_temp.index, target_worker);
} else {
    gthreadpool.set_worker_busy(target_worker, true);  // 标记占用
    // Path C: RGA 直写 NPU 物理内存
    // ...
}
```

### Worker 侧的释放（thread_pool.cpp:204）

```cpp
// 推理完成后释放
worker_ctx[id].is_busy.store(false);
```

### 时序关系

```
ReadVideo:               Worker:
  is_busy? → false         推理开始
  标记 is_busy=true         ...
  Path C 写入 NPU          rknn_run（NPU 读已写入的数据）
  入队 g_SafeQueueRead      rknn_run 完成
  ─────── 下一帧 ──────→   后处理 + 绘制
  is_busy? → true!         结果入 result_queue
  skip_inference = true    is_busy.store(false)  ← 释放
  ─────── 再下一帧 ────→   下一帧推理...
  is_busy? → false
```

### 时序窗口问题

`is_busy` 不是同步原语，存在时序窗口：

```
ReadVideo                     Worker
  │                             │
  │ ← 检查 is_busy = false      │
  │                             │ rknn_run 开始 → is_busy = true
  │ Path C 开始写 NPU（踩窗口）   │
  │                             │
```

此时 ReadVideo 踩在窗口里写了 NPU 数据，但 Worker 已经开始读旧数据推理。结果：**画面花屏或推理结果异常**。

**实际影响：** 概率极低（窗口 ~纳秒级，帧间隔 33ms）。且下一帧通过 `skip_inference` 正常处理。如果特别关注此问题，可在 ReadVideo 中写 NPU 后再次检查 `is_busy`，但这引入了额外开销。

## 5.9 in_flight_count < 20 限流分析

**位置：** `thread_pool.h:126`（定义）+ `main.cpp:333`（使用）

### 计数器维护

```cpp
// thread_pool.cpp:91 — 提交时 +1
submit_to_worker() { tasks_in_flight++; ... }

// thread_pool.cpp:201 — Worker 完成时 -1
worker() { ... tasks_in_flight--; }
```

### count() 实现

```cpp
// thread_pool.h:94
int in_flight_count() { return tasks_in_flight.load(); }
// 使用 memory_order_seq_cst（默认），保证读到最新值
```

### 什么情况下会达到 20

**稳态（30fps）：** `in_flight_count = 4`（每 Worker 1 帧），永远不会碰到 20 天花板。

**场景一：下游反压解除后突发回填**

```
RTMP 网络抖动 → WriteVideo 推流阻塞
  → g_SafeQueueWrite 填满（100）
    → Aggregator 步骤 C 的 enqueue() 阻塞（条件变量挂起）
      → 提交循环（步骤 A）也不再执行
        → ReadVideo 继续生产，g_SafeQueueRead 积累帧

反压解除（~1s 后）：
  → g_SafeQueueWrite 有空间
  → Aggregator 的 enqueue 返回
  → 回到步骤 A：g_SafeQueueRead 积了 ~30 帧
  → while(!empty && in_flight < 20) 一口气提交 20 帧
  → in_flight_count = 20 ← 达到上限！
  → 暂停提交，先收集结果、写出、推进 nextWriteIndex
  → 再回来 drain 剩下的帧
```

**场景二：系统启动瞬态**

启动时 Workers 还在初始化（模型加载、NPU 预热），`rknn_run` 第一次调用可能耗时 50-100ms。同时 ReadVideo 已在 30fps 生产，Aggregator 提交速度 > Worker 消费速度，`in_flight_count` 短暂攀升到 20。

**场景三：推理耗时暴涨**

要让 `in_flight_count` 到达 20，需要 `20 × frame_interval < inference_time`：

| 帧率 | 需要推理耗时 > |
|------|--------------|
| 30fps | 20 × 33ms = **660ms** |
| 60fps | 20 × 17ms = **340ms** |

正常 YOLOv5s 推理 ~20ms，不会触发。但极端情况（画面极复杂、NMS 目标数暴增）可能叠加场景一共同导致。

### 20 限流的真正作用

**控制 result_buffer 峰值，从而控制步骤 C 对 g_SafeQueueWrite 的写入节奏。**

如果没有限流，最坏情况推演（考虑 Core 0 串行瓶颈）：

```
g_SafeQueueRead 积压 100 帧 → 全部提交
  → Core 0 (W0+W3): 50 帧 × 20ms = 1000ms 消化 ← 瓶颈
  → ~500ms 后: W1+W2 的 50 帧完成 → result_buffer 堆积 ~50 帧
  → ~1000ms 后: 全部 100 帧完成 → result_buffer 峰值 100 帧
  → 步骤 C 顺序写出，但 g_SafeQueueWrite 容量 100
  → 如果 WriteVideo 稍慢，enqueue 阻塞 → Aggregator 卡死
```

有限流 20 时：

```
g_SafeQueueRead 积压 100 帧
  → 只 drain 20 帧提交 → 200ms 后 20 个结果回到 result_buffer
  → 步骤 C 写出 → nextWriteIndex 推进
  → 回到步骤 A，再 drain 20 帧...
  → 每批 20 帧，共 5 批消化完
```

| 指标 | 无限制（100 帧一次） | 有限流（20 帧一批） |
|------|--------------------|--------------------|
| result_buffer 峰值 | **~100 帧** (~300MB+) | **~20 帧** (~60MB) |
| 最大帧延迟 | **~1000ms**（30 帧 @30fps） | **~200ms**（6 帧） |
| 反压恢复时间 | **~1000ms** | **~200ms** |

### 为什么不设更大或更小

- `20 ÷ 4 Worker = 5 帧/Worker`，最末帧等待 `5 × 20ms = 100ms`，延迟可接受
- `20 × ~3MB/帧 = ~60MB`，RK3588 8GB 合理
- 设太小（如 8）：频繁进出提交循环，批处理效率降低
- 设太大（如 100）：限流形同虚设

## 5.11 线程退出顺序

```
STREAMON → ReadVideo 退出 → Aggregator 退出 → WriteVideo 退出
              │                    │                    │
          g_readFinish=true    g_processFinish=true    break
```

**原子变量传递链：**

```
ReadVideo (采集结束)
  → g_readFinish = true
    → Aggregator 检测到 + g_SafeQueueRead 空 + in_flight=0 + result_buffer 空
      → break → g_processFinish = true
        → WriteVideo 检测到 + g_SafeQueueWrite 空 → break
```

**SafeQueue 的 `stop()` 是兜底机制**（`main.cpp:643-644`），在线程 join 前调用，确保阻塞在 `dequeue` 上的线程能被唤醒退出：

```cpp
g_SafeQueueRead.stop();
g_SafeQueueWrite.stop();
```

`stop()` 设置 `stop_flag = true` 并 `notify_all()`，所有条件变量 wait 立即返回。之后线程 join 即可安全退出。

## 5.12 与旧架构的关键对比

| 对比项 | 旧架构（面试基准） | 新架构（当前代码） |
|--------|------------------|------------------|
| 提交方式 | `submit_task_async()` → 返回 `future<ProcessResult>` | `submit_to_worker(id, ...)` → 无返回值 |
| 结果回传 | `promise.set_value()` + `future.get()` | `result_queue.push()` + `try_get_result()` |
| 任务队列 | 全局 `SharedQueue<Task>`（4 Worker 竞争锁） | 每 Worker 独立 `queue<Task>`（无竞争） |
| 帧保序 | `map<int, future>>` + `wait_for(0)` 轮询 | `result_buffer` map + `try_get_result` 批量 drain |
| 阻塞模型 | `future.get()` 可能阻塞（旧架构 future 语义） | 三步循环：提交/收集/写出，各环节非阻塞 |
| 限流 | `tasks_inflight.size() < 20` | `in_flight_count() < 20`（语义相同，实现不同） |
| 丢帧 | 无 | `is_busy` atomic + `skip_inference` 丢帧 |
| NV12 内存 | ReadVideo 中 `malloc+memcpy` 1 次 + Worker 中 NV12→RGB 1 次 | 仅 Worker 输出 `malloc` 1 次（BGR→NV12） |
