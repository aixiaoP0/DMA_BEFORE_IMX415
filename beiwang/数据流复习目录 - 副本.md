# 项目数据流复习目录

按数据在硬件和软件之间的流转路径，从上到下逐层深入。

```
IMX415 Sensor
    ↓ RAW10 (Bayer)
MIPI CSI-2 4-lane
    ↓
ISP (RKAIQ)
    ↓ YUV420 (NV12)        ← ISP 硬件完成 RAW→YUV
V4L2 MPLANE + DMABUF
    ↓
Thread_ReadVideo
    ├→ Path A: RGA NV12→BGR           ← OpenCV 画框用
    └→ Path C: RGA NV12→RGB→NPU fd    ← 直写 NPU 物理内存（零拷贝核心）
    ↓ (SafeQueue)
Aggregator
    ↓ (submit_to_worker + result_queue)
Worker[i]
    ├→ NPU: YOLOv5s 推理               ← 输入已被 ReadVideo 通过 RGA 填满
    ├→ CPU: 后处理（反量化 + NMS）← output_mems->virt_addr
    ├→ CPU: draw boxes
    └→ RGA: BGR→NV12       ← 编码器输入
    ↓ (SafeQueue)
Thread_WriteVideo
    ├→ MPP Encoder: H.264
    └→ RTMP Stream
```

## 第一阶段：Sensor 数据采集（硬件层）

### 1.1 IMX415 传感器输出 RAW10

- **硬件的启动时序**：IMX415 传感器上电后，需要通过 I2C 总线写入寄存器配置（分辨率、帧率、曝光模式等）
- **RAW10 格式**：每个像素 10bit，Bayer 模式（RGGB 排列），Sensor 输出的是最原始的"未经过任何处理"的光电信号
- **MIPI CSI-2 4-lane**：4 对差分信号线同时传输，每条 lane 约 1.2Gbps，总带宽约 4.8Gbps，足以承载 1280x720@30FPS RAW10 数据
- **D-PHY 物理层**：D-PHY 负责串行/解串、时钟恢复、通道对齐

**相关文件：** `beiwang/Untitled-1.dts`
**关键知识点：** MIPI CSI-2 协议层（长包/短包、ECC 校验）、D-PHY 状态机、Bayer 格式

### 1.2 ISP 硬件管线（RAW→YUV）

- **Demosaic（去马赛克）**：Bayer 每个像素只有一种颜色，通过周围像素插值恢复 R/G/B 三个通道
- **3A 算法（RKAIQ）**：自动曝光（AE）→ 调整曝光时间/增益；自动白平衡（AWB）→ 调整 R/G/B 增益；自动对焦（AF）→ 调整镜头位置（定焦模组可能无此功能）
- **色彩校正矩阵（CCM）**：将 Sensor 的 RGB 色彩空间映射到标准 sRGB
- **Gamma 校正**：非线性映射，适应人类视觉特性
- **YUV420（NV12）输出**：Y 平面 + UV 交错平面。`YSize = W*H`，`UVSize = W*H/2`，`Total = W*H*3/2`

**知识点：** ISP 管线各模块作用、3A 算法基础、NV12 内存布局

## 第二阶段：V4L2 视频采集（设备层）

### 2.1 设备节点与格式协商

- **设备节点** `/dev/video11`：RK3588 ISP 主通道（mainpath），另一路 `/dev/video12` 是自拍通道（selfpath）
- **V4L2 MPLANE 多平面 API**：Rockchip ISP 驱动强制要求 `V4L2_BUF_TYPE_VIDEO_CAPTURE_MPLANE`
- **格式协商流程**：`VIDIOC_S_FMT` 设置格式 → `VIDIOC_REQBUFS` 申请缓冲区 → `VIDIOC_QBUF` 入队 → `VIDIOC_STREAMON` 启动

**相关代码：** `main.cpp:474-590`
**核心代码行：** `main.cpp:496-508`

```cpp
fmt.fmt.pix_mp.pixelformat = V4L2_PIX_FMT_NV12;
```
**追问：** 为什么 IMX415 用 NV12 而不用 NV16 或 RGB？→ NV12 是 YUV420 子采样，人眼对亮度敏感对色度不敏感，颜色信息减半但人眼几乎看不出差异，节省带宽和存储

### 2.2 DMABUF 缓冲区池

- **dma_heap 分配器**：`/dev/dma_heap/` 下保留的连续物理内存区域，通过 ioctl `DMA_HEAP_IOCTL_ALLOC` 分配
- **预分配 4 个缓冲区**：为什么是 4 个？→ 保证在 3 个正在使用的状态下总有 1 个空闲可写入，避免采集停顿
- **mmap 映射**：将物理 DMA 内存映射到用户空间虚拟地址，CPU 可通过 `mapped_addr` 访问

**相关代码：** `dmabuf.h/c`、`main.cpp:531-559`
**结构体：** `dmabuf_heap_t`（堆描述符）、`dmabuf_buffer_t`（缓冲区描述符）

**数据流：**
```
ISP 硬件写入 DMABUF → V4L2 出队 → CPU sync_start → CPU 读取 → sync_stop → V4L2 入队
```

### 2.3 Poll + DQBUF 采集循环

- **poll 机制**：注册 `POLLIN` 事件，等待内核通知帧就绪，超时 5 秒
- **DQBUF 出队**：从 V4L2 驱动取出已填满数据的缓冲区索引
- **DMABUF fd**：通过 `my_dmabuf_get_fd()` 获取 fd，可直接传给 RGA 硬件操作，无需 CPU 拷贝

**相关代码：** `main.cpp:169-295`（`Thread_ReadVideo` 完整函数）

## 第三阶段：RGA 格式转换（ReadVideo 线程）

### 3.1 第一条路径：NV12→BGR888（OpenCV 画框）

- **输入**：DMABUF fd（ISP 已写入 NV12 数据）
- **输出**：OpenCV `cv::Mat` BGR888 格式
- **硬件路径**：RGA 从 DMABUF fd 直接读取→内部 CSC 色彩空间转换→写入 CPU 内存
- **耗时**：约 3ms（纯 CPU 实现需 ~30ms）

**相关代码：** `main.cpp:233-265`

```cpp
rga_buffer_handle_t src_handle = importbuffer_fd(buffer_info.dmabuf_fd, nv12_size);
// 关键：从 DMA fd 导入，RGA 硬件直接读
rga_buffer_t src = wrapbuffer_handle(src_handle, width, height, RK_FORMAT_YCbCr_420_SP);
rga_buffer_t dst = wrapbuffer_handle(dst_handle, width, height, RK_FORMAT_BGR_888);
imcvtcolor(src, dst, RK_FORMAT_YCbCr_420_SP, RK_FORMAT_BGR_888);
```

### 3.2 Path C：RGA 硬件直写 NPU 物理内存（取代 NV12 拷贝）

- **重构后不再有 CPU 参与的 NV12 拷贝**：DMABUF 中的数据通过 RGA 硬件 DMA 直接读取，缩放+色彩转换后直写 NPU 物理内存
- **核心代码**：`importbuffer_fd(DMABUF fd)` → `imresize(NV12→RGB 640x640)` → `importbuffer_fd(NPU fd)`
- **Round-Robin 调度**：ReadVideo 通过 `index % g_num_workers` 选定目标 Worker，获取该 Worker 的 NPU 输入内存 fd
- **`is_busy` 保护**：如果目标 Worker 的 NPU 缓冲区正被占用（仍在推理上一帧），触发 `skip_inference=true` 主动丢帧

**相关代码：** `main.cpp:249-274`

```cpp
int target_worker = frame_temp.index % g_num_workers;
int npu_fd = gthreadpool.get_worker_input_fd(target_worker);

rga_buffer_handle_t src_c = importbuffer_fd(buffer_info.dmabuf_fd, nv12_size);
rga_buffer_handle_t dst_c = importbuffer_fd(npu_fd, 640 * 640 * 3);
// RGA 硬件 DMA 直接完成：缩放（1280x720→640x640）+ 色彩转换（NV12→RGB）+ 写入 NPU
imresize(src, dst);
```

**追问：** 为什么不在 ReadVideo 中拷贝 NV12 给 Worker 用？→ 重构后 RGA 在 ReadVideo 中直接完成缩放+色彩转换+写入 NPU 物理内存，Worker 只需要触发 `rknn_run`，NV12 数据不再需要经过 CPU 堆内存中转。一次 RGA DMA 操作替代了 CPU `memcpy` + Worker RGA 两步。

## 第四阶段：线程间同步（SafeQueue）

### 4.1 有界阻塞队列

**相关代码：** `SafeQueue.h`

| 操作 | 阻塞条件 | 唤醒条件 |
|------|---------|---------|
| `enqueue` | 队列满（`size >= maxSize`） | `not_full`（有元素出队） |
| `dequeue` | 队列空 | `not_empty`（有新元素入队） |

**条件变量 wait 的虚假唤醒防护：**
```cpp
cond_not_full.wait(lock, [this]{ return q.size() < maxSize; });
// 等价于：
while (q.size() >= maxSize) cond_not_full.wait(lock);
```

**追问：** 为什么用 `notify_all` 而不是 `notify_one`？→ 当多个生产者都在等待入队空间时，出队一个元素可能刚好让多个生产者都能入队，`notify_all` 避免死锁风险

## 第五阶段：Aggregator 调度与保序

### 5.1 聚合线程调度逻辑

**相关代码：** `main.cpp:298-380`（`aggregatorThreadFunc`）

**两个循环：**
1. **提交循环**（lines 314-327）：从 `g_SafeQueueRead` 批量取帧，提交到线程池，存入 `map<index, future>`
2. **收集循环**（lines 331-363）：检查 `nextWriteIndex` 对应的 `future` 是否完成，完成则取出结果入 `g_SafeQueueWrite`

**保序机制：**
```
tasks_inflight = {
    0: future<ProcessResult>,    // 已提交，可能已完成
    1: future<ProcessResult>,    // 已提交，可能已完成
    2: future<ProcessResult>,    // 已提交，可能已完成
    3: future<ProcessResult>,    // 已提交，推理中
    ...
}

nextWriteIndex = 0  // 只检查 index==0 的 future，完成后 +1 检查 index==1
```

**追问：** 为什么用 `std::map` 而不是 `std::unordered_map`？→ map 是有序的，`find(nextWriteIndex)` 存在则说明该帧还在 inflight。如果某个帧的推理结果被跳过了（比如 index 5 完成后 index 4 还没完成），map 能保持有序。但这里其实更关键的语义是顺序遍历，map 的 `O(log n)` 查找完全够用（最多 20 个元素）

**追问：** 如果某一帧推理特别慢（比如 index 5），会怎样？→ 后面的帧即使先完成也必须等待 index 5，造成 Head-of-Line blocking。这是保序的代价。可通过超时机制或跳帧策略缓解

## 第六阶段：Worker 线程池推理

### 6.1 线程池与 YOLO 模型初始化

**相关代码：** `thread_pool.h/cpp`

- **4 个 Worker**，每个绑定到 CPU4-7 大核（`pthread_setaffinity_np`）
- **4 个 YOLOv5s 实例**，每个绑定到不同 NPU 核心（`i % 3` 轮询核 0/1/2）
- **每个实例独立的 RKNN 上下文**：`rknn_init` + `rknn_create_mem` 分配 NPU 物理内存

**追问：** 为什么需要 4 个模型实例而不是 1 个加锁？→ RKNN 上下文不是线程安全的。如果 4 个线程共享一个实例，推理任务会串行化，相当于单核性能

### 6.2 重构说明：NV12→RGB888 已移至 ReadVideo Path C

- **旧版**：Worker 从 CPU 堆内存读取 NV12 拷贝（`importbuffer_virtualaddr`），RGA 缩放+转换后写入 NPU 物理内存
- **新版**：ReadVideo 线程直接从 DMABUF fd 读取 NV12，RGA 缩放+转换后直写 NPU 物理内存（`importbuffer_fd`）。Worker 只需触发 `rknn_run`
- **重构收益**：消除了 `malloc+memcpy` 和 `importbuffer_virtualaddr`，实现 ISP→RGA→NPU 全链路零拷贝

**相关代码：** `main.cpp:249-274`（ReadVideo Path C）

### 6.3 NPU 零拷贝推理

**相关代码：** `yolov5s.cpp`

**零拷贝链路：**
```
rknn_create_mem(ctx, size)      → 分配 NPU 侧物理内存
rknn_set_io_mem(ctx, mem, attr) → 绑定输入/输出到 RKNN 上下文
RGA 写入 input_mem              → 硬件直接写 NPU 内存
rknn_run(ctx, NULL)             → NPU 直接读 input_mem 推理
output_mems[i].virt_addr        → CPU 直接读推理结果（INT8）
```

**追问：** 不零拷贝会怎样？→ 每次推理前需要 `memcpy` 输入数据到 NPU 内存，推理后再 `memcpy` 结果回 CPU 内存。两张额外的内存拷贝在 30FPS 下就是每秒 60 次拷贝

### 6.4 INT8 后处理与 NMS

**相关代码：** `post_process.h/cpp`

**反量化：** `float_val = (int8_val - zp) * scale`

**三尺度解码（8x/16x/32x 下采样）：**
| 输出分支 | 下采样倍数 | 网格大小 | 负责目标 |
|---------|-----------|---------|---------|
| output0 | 8x | 80x80 | 小目标 |
| output1 | 16x | 40x40 | 中目标 |
| output2 | 32x | 20x20 | 大目标 |

**NMS 流程：**
1. 置信度过滤（`box_threshold = 0.5`）
2. 按类别分组
3. 单个类别内按置信度排序
4. 从最高分开始，IOU > 0.5 的框被抑制

**追问：** 为什么两个阈值都设置 0.5？→ 0.5 是 COCO 数据集上的常用平衡值。太低产生很多误检（FP），太高漏检（FN）严重。

### 6.5 后处理 → 绘制 → BGR→NV12

**后处理（NPU 输出 → CPU 读取）：** `post_process.h/cpp`
- NPU 推理结果通过 `output_mems[i]->virt_addr` 由 CPU 读取（INT8 格式）
- 反量化：`float_val = (int8_val - zp) * scale`
- 三分支解码 + NMS → `detect_result_group_t`（最多 64 个检测框）

**关键理解：** NPU 输出的是**检测元数据**（坐标、类别、置信度），不是像素数据。因此不存在"NPU→编码器直接通路"，元数据必须走 CPU 解析。

**绘制：** `yolov5s.cpp:282-304` — `cv::rectangle` + `cv::putText` 将检测框画到 BGR 图像上

**第三条路径（BGR→NV12，Worker 输出）：**
- 绘制后的 BGR Mat → `BGR_to_NV12_with_rga()` → NV12 数据
- NV12 是 MPP 编码器要求的输入格式

**相关代码：** `thread_pool.cpp:199-217`

## 第七阶段：编码与推流（WriteVideo 线程）

### 7.1 MPP 硬件编码

- **MPP（Media Process Platform）**：Rockchip 的媒体处理硬件
- **输入**：NV12 裸数据
- **输出**：H.264 压缩码流
- **CPU 占用率**：约 5%（纯软件编码器如 x264 可达 50-80%）

**相关代码：** `streamer.h/c`、`mpp.c/h`

- **推流地址**：`rtmp://192.168.15.214:1935/live/cv`
- **延迟**：<200ms（MPP 编码延迟 + 网络传输 + 服务器转发）

**相关代码：** `rtmp.h/c`

### 7.3 FPS 统计

- 每 30 帧计算一次
