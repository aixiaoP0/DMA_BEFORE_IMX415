# xv6 实验面试回答整理

## 1. 简化用户空间和内核空间之间传递数据的流程

原来传递数据，比如说从内核模式下想访问用户空间的地址，我们需要手动模拟MMU访问页表项的过程，找到对应的物理地址。那我们就提出疑问，为什么内核态MMU无法访问用户空间地址对应的物理内存呢？其实在 RISC-V 中，一旦 `satp` 寄存器写入了有效页表基址，S-mode 下 CPU 发出的所有访存地址都会被 MMU 自动翻译，不存在"某些地址绕过 MMU"的情况。原本设计的症结就是内核页表没有用户地址空间的映射。那为什么没有呢？因为在原本的设计中，只有一个全局内核页表，若把 0x1234 映射成进程 A 的物理页 PA1，进程 B 再进入系统调用、也访问 0x1234 时，内核页表映射就会冲突错乱。

所以我们的大体工作就是为每个进程分配独立的内核页表和内核栈，然后在用户进程初始化时，把用户页表的虚拟地址到物理地址的映射，以相同的虚拟地址但在内核页表中建立映射并清除 `PTE_U` 使 S-mode 可访问。然后设置在调度时，切换到进程的内核页表；切回时恢复全局内核页表。

## 2.1 实现内存页面的懒分配

原始的 sbrk 系统调用在扩展堆内存时，会立即分配物理内存并建立页表映射，但其实许多程序申请的内存远超实际使用，并且大块分配会阻塞进程很长时间。所以我们的核心思路是在 sbrk 系统调用时，仅修改进程堆内存边界地址，不分配物理内存也不建立页表映射，然后在访问新地址时，利用缺页故障触发，此时会检查出错的原因，通过 `stval` 寄存器拿到出错的虚拟地址，检查地址属于懒分配范围后，再进行物理内存分配和映射。

具体的检查条件：地址必须在用户栈之上，地址必须在进程堆范围内，页表项确实无效。此外还需要修改各处 panic 为跳过不存在的映射，因为现在存在合法的缺页情况。

## 2.2 实现写时复制

原始的 fork 完整复制父进程的所有物理页给子进程，时间开销大，如果父子进程都是对这块内存只读，那根本就不需要重复存储。我们的修改就是共享物理页，写时才复制。主要分三个核心环节：

首先是 fork 阶段，修改了原来的 `uvmcopy` 函数。现在它不再为子进程分配物理内存，而是直接把父进程的物理地址映射过去。为了实现写时复制，把父子进程对应的 PTE 全部设为不可写，并利用 PTE 里的保留位（RSW）打上一个 COW 标记。同时维护了一个全局的引用计数数组 `ref_count`，每当一个物理页被多映射一次，计数就加 1。

当进程尝试写这些只读页面时，硬件会触发一个 Store Page Fault。内核捕获到异常后，先通过 `stval` 寄存器拿到出错的虚拟地址，检查对应的 PTE 是否带有 COW 标记。如果是，就去查这个物理页的引用计数。这里有个优化点：如果引用计数大于 1，说明还有其他进程在用，就分配一个新的物理页，把数据拷贝过去，然后更新当前进程的页表指向新页，并把权限改回可写，原页面的计数减 1；但如果引用计数已经等于 1 了，说明只有当前进程在用，就直接把这个页面标记为可写，省掉了一次内存分配和拷贝的开销。

最后在内存释放这块，重写了 `kfree`。现在调用 `kfree` 不会直接回收内存，而是先让引用计数减 1，只有当计数归零时，才真正把这块物理内存放回空闲链表。为了保证多核下的安全，操作计数数组时加了自旋锁。

## 3. 每CPU空闲链表与局部锁策略

原本分配物理内存的 kalloc 使用 freelist 链表，将空闲物理页本身直接用作链表项连接成一个链表。分配时将物理页从链表移除，回收时放回。分配和释放都是操作共享数据，修改 freelist 链表，因此为了线程安全都加上了锁，导致同一时刻只能有一个线程申请分配或释放内存。

我们的做法是将原本的一个全局 freelist 加一把大锁，拆分为每 CPU 一个独立的 freelist 加每 CPU 的一把小锁。这里需要注意的情况就是可能一个 CPU 对应的空闲物理页不够了，需要去偷其他 CPU 的页，避免死锁是此实现的难点。

当当前 CPU 的 freelist 已经空了，我们去其它 CPU 偷内存的时候，先释放本地锁，然后准备一个不用加锁的临时链表头，遍历其他 CPU，加上对方的锁，把页偷到临时链表里，然后立刻释放对方的锁。拿着装满页的临时链表，重新上本地锁，检查当前 CPU 是否有空闲页——因为我们没拿锁的这段时间里，假设有个 kfree 操作刚好把某些页放进了我们当前 CPU 的 freelist。所以我们将原链表挂载到偷来链表的尾部，然后更新当前 CPU 的链表头指针。

## 4. 基于内部中断机制实现 alarm 系统调用

在用户态触发 trap，硬件自动完成切换特权级，设置 sepc（供返回用户态时程序继续执行），设置 scause 寄存器（trap 的原因），跳转到 stvec（陷阱向量寄存器）保存的地址。执行 uservec 用户态陷阱向量入口对应的汇编代码，保存所有用户寄存器到 trapframe，加载 trapframe 头部预存的内核上下文，切换到内核页表和内核栈，刷新 TLB，跳转到 `usertrap()`。判断 scause 分流处理，如果是系统调用执行 `syscall`。在完成对应的逻辑后，调用 `usertrapret()`，在这里往 trapframe 中填入下次陷入时内核所需的信息（内核页表、内核栈顶、cpuid 等），对系统调用设置 sepc+4，调用 `userret`，切换到用户页表，从 trapframe 恢复所有用户寄存器并返回用户态。

这里的 alarm 调用就是为用户程序提供两个系统调用：`sigalarm`，设置一个定时器，每 ticks 个时钟中断后内核自动调用用户提供的 handler 函数；`sigreturn`，在 handler 中调用，恢复用户程序原先的执行流。

设计 alarm 面临的核心挑战有三点：第一，定时器是在时钟中断处理中也就是内核态检测到期的，但 handler 是用户态的函数，如何强制内核跳转到用户态函数？第二，handler 执行完毕后，如何精确恢复到中断的程序位置，就像什么也没有发生？第三，防止重入，如果在 handler 执行期间再次发生时钟中断并尝试调用 handler，就会导致状态混乱。

具体实现，我们在进程初始化时添加了一个独立的物理页，用于备份正常的 trapframe。然后在进程结构体里加了几个字段：interval 用来存用户设置的 ticks 数，handler 存用户传进来的函数指针，ticks 做计数器，再加一个 alarmed 标志位防止重入。sigalarm 的实现很简单，把用户传的 interval 和 handler 存进进程结构体，重置 ticks 和 alarmed 就行。

关键是 usertrap 里怎么处理。每次时钟中断的时候 ticks 加一，当 ticks 达到 interval 并且 alarmed 为 0，说明该调 handler了。这时候先把当前的 trapframe 整个拷贝到刚才那个备份页里，然后把 trapframe->epc 改成 handler 地址，设 alarmed 为 1。这样中断返回的时候 sret 就会跳到 handler 执行。为什么能这么干？因为 trapframe 里存的就是用户态被中断时的全部寄存器状态，epc 指向中断发生时正在执行的指令，改了 epc 就等于改了返回地址。

那 handler 执行完怎么回去呢？这就是 sigreturn 干的事。handler 调用 sigreturn，sigreturn 把备份的 trapframe 拷回来，alarmed 清 0，然后正常返回。这时候 epc、所有寄存器都恢复成中断前的样子，程序就像啥也没发生一样继续跑。

重入问题就是靠那个 alarmed 标志。handler 跑的时候 alarmed 是 1，这时候再来时钟中断，检查到 alarmed 为 1 就不会再备份和跳转，直接 yield。只有 sigreturn 把 alarmed 清 0 之后才能再次触发。

## 5. 实现内存映射文件

在进程结构体里加了一个 VMA 数组，每个 VMA 就是一段虚拟地址到文件的映射记录，包含起始地址、长度、权限、MAP_SHARED 还是 MAP_PRIVATE、对应的 file 结构体指针、文件内偏移。进程最多同时映射 16 个区域，数组就够了。

mmap 的实现主要就是检查参数合法性，然后在进程的地址空间里找一块空闲区域，我选择从 TRAPFRAME 往下生长的位置，毕竟堆是从下往上长的、栈也是从高地址往下，mmap 区域再往下放，互不干扰。找到空闲区间后就把这段信息填进一个 VMA 条目，映射关系的记录工作就完了。注意这里要调用 `filedup` 增加文件的引用计数，否则用户关闭文件描述符后 file 结构体可能被释放。我们采用懒加载策略，一页物理内存都不分配，文件内容也不读。返回映射的起始地址就行。

那文件内容什么时候真正读进来呢？缺页异常的时候。在 usertrap 里处理 page fault 时，除了检查 COW、懒分配，还要检查出错的地址是不是落在某个 VMA 范围内。如果是，说明这是 mmap 之后第一次访问这一页。这时候才 kalloc 一个物理页，用 `readi` 从磁盘把文件对应位置的内容读进来，然后把这页映射到用户地址空间。如果是 MAP_SHARED 的映射，还要在页表项里设个脏页标记，这样后面才知道哪些页被改过、需要写回磁盘。

munmap 就正好反过来。先查 VMA 找到要解除的映射区域，然后遍历这个区域里的每一页。对于 MAP_SHARED，检查哪些页是脏的，用 `writei` 把它们写回磁盘，然后逐页 unmap、释放物理页。最后调用 `fileclose` 释放文件引用，把 VMA 条目清掉。

还有进程退出的时候，exit 里要遍历所有 VMA，把还没 munmap 的区域全部清理一遍，不然文件修改就丢了。另外 fork 时也需要继承父进程的 VMA 数组，子进程的每个 VMA 都要重新 `filedup` 来持有文件引用。

说到底，mmap 这个功能在操作系统里看起来很复杂，但在 xv6 上做起来其实就是在缺页处理这个钩子上挂一段逻辑——文件内容不需要提前加载，等真的访问那一页的时候再按需从磁盘读进来，改过的页等 munmap 或者进程退出的时候再写回去。这就是典型的 demand paging 思想，把真正的数据搬运推迟到最后一刻。

## 6. 添加系统调用

先在 `kernel/syscall.h` 里加个编号，给新系统调用分配一个数字，比如 `#define SYS_mysyscall 22`。这个编号就是个约定，用户态用这个号告诉内核自己想调哪个功能。

然后编写实际干活的函数，比如 `sys_mysyscall(void)`，参数用 `argint`、`argaddr` 这些函数从用户栈上取。

接着在 `syscall.c` 中添加 extern 声明，在系统调用函数指针表 `syscalls` 里注册上：`[SYS_mysyscall] sys_mysyscall`。

用户态这边得加个入口，写一段汇编，做的事情就是把系统调用号装到 a7 寄存器里然后 ecall。这就是 `usys.pl` 生成的汇编 stub。
